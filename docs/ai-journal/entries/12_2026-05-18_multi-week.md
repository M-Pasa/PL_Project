# Entry 12 — `plan N week` extension across Parser, AST, Interpreter

## Goal

Light up the multi-week loop that entry 11's interpreter left stubbed at N=1.
Extend `<schedule-stmt>` to accept the optional `NUM_LIT` already present in
the §4.4.8 spec, project bodyweight per week with the goal rate, and apply
each workout's `progress weight r` to its exercises so weeks 2..N are
observationally distinct from week 1.

## Prompt

Continuation prompt: "Let's continue from where we left." I read the previous
session's tail to recover state, then asked which deferred item to tackle —
`plan N week`, §4.7 expression-bodied slots, §4.8 design rationale, or quiz
prep. The user picked `plan N week extension`.

## Response (key part)

Five edits, each scoped to one concern:

- `Ast.ScheduleStmt` gains `int weeks` (default 1 when the `NUM_LIT` is
  absent).
- `Parser.parseScheduleStmt` peeks for `NUM_LIT` between `plan` and `week`;
  reuses `parsePositiveInt` so `plan 0 week …` becomes a parse error.
- `AstDumper.dumpScheduleStmt` prints the `weeks` line for `--dump-ast`.
- `Interpreter.run` materialises `N` `Week` values; for each `k` it computes
  `offsetSec = (k-1) × 604800` and projects `bw_k = bw₀ + bwChangePerSec ×
  offsetSec`, where `bwChangePerSec` is `±goalRate.v` with the sign drawn
  from `Direction.LOSE`.
- `evalWorkout` now takes `offsetSec` and pulls `wkPerSec` from
  `w.progress` (and `rkPerSec` from `rd.progress` for routine calls). The
  new `evalExercise(e, progressGPerSec, offsetSec)` returns
  `MassV(base_g + progressGPerSec × offsetSec)`.

Verified on `samples/valid_04_multi_week.fl` (4-week bulk):
`bw 78 → 78.5 → 79 → 79.5 kg`, Bench `80 → 87.5 kg`, per-bw target threshold
`140.4 → 143.1 g`. Sign flip checked with a `cut`/`lose 0.5 kg/week` snippet:
`78 → 77 kg` over three weeks.

## Accepted

- Sign-of-direction lives at one place: a single `int sign` computed once
  from `plan.goalRate.direction`, fed into `bwChangePerSec`. The interpreter
  never re-inspects `Direction` inside the per-week loop.
- Progression carried as `progressGPerSec` (canonical g/s) and offset as
  `offsetSec`. Both already canonicalised by `liftRate`/`toCanonical`, so the
  per-week arithmetic is plain doubles — no per-iteration unit lookups.
- `parsePositiveInt` reused as-is for the `NUM_LIT` between `plan` and `week`.
  That gives `plan 0 week …` and `plan 2.5 week …` the same diagnostic shape
  as the existing `sets:`/`reps:` validation.

## Rejected / modified

- Considered threading the week index `k` into `evalWorkout`/`evalExercise`
  and recomputing the progression delta as `r × ((k-1) week)` symbolically
  inside the value layer. Rejected — `offsetSec` is already canonical and the
  multiplication is one line; the symbolic route would have re-derived the
  same number through `liftFromFamily`. Kept the doubles path consistent with
  the rest of the interpreter.

## Errors I caught

- **Routine-level `progress`.** First sketch only pulled progression from
  `w.progress`. Grepping `progress|Progress` against the AST surfaced
  `RoutineDecl.progress` as a separate field — without that pass the routine
  branch of `evalWorkout` would have silently skipped progression for any
  exercise reached through `use "R"(...)`. Fixed by computing `rkPerSec` from
  `rd.progress` before binding the params.
- **`plan 0 week`.** Almost let `weeks` be any `int`. Recognised that
  `parsePositiveInt` (already used for `sets`/`reps`) is the exact validator
  I want — rejects 0, rejects fractions, with the existing diagnostic
  format.
- **Backward compatibility.** Worried briefly that the bare `plan week …` in
  the existing samples would break. Verified by running all four
  `samples/valid_*.fl` through `Main` after the change — they still parse,
  type-check, and (where applicable) interpret unchanged.

## Reflection (2–3 sentences)

The structural prep in entry 11 paid off: the per-week loop was already
shaped right, so the change collapsed to "feed it a real N and an
`offsetSec` term." The one place I almost lost a feature was routine
progression — a five-second grep saved a silent bug, which is the lesson:
when a field exists in two parallel AST nodes, the interpreter must touch
both.
