# Entry 13 — §4.7 expression-bodied slots and the prefix let-section

## Goal

Resolve the "let bindings are observationally dead" gap entry 11 flagged.
Widen the six value-position slots from `<quantity>`/`<rate>` to
`<expression>` so `let`-bound names can finally be referenced inside
workouts, meals, targets, and goal rates. Solve the implied scoping problem
along the way.

## Prompt

After we summarised the P2 status, the user picked "let's finish 4.7." Before
writing code I asked for a design decision because §4.6.3 and §4.6.5 are in
mild tension on the let-into-workout question:

- §4.6.3 rule 1: a let is visible only "in every expression that follows
  it" within the same athlete-block.
- §4.6.5 last bullet: in P2, workout/meal bodies should see athlete-scope
  lets. But the grammar puts the rules-section (where lets live) *after*
  workouts and meals.

Offered three resolutions; user picked **"Move lets before bodies"** — a
new prefix `<let-section>` between routine declarations and the body
sections. The "follows" rule stays literal, the single-pass type check
stays intact, and `let` stops being a `<rule-decl>` alternative.

## Response (key part)

Six slots widened to `<expression>` (parser side):

- `<exercise-decl>` `weight:`
- `<macro-decl>` value (protein/carbs/fat)
- `<routine-call>` arguments
- `<target-decl>` threshold
- `<progress-stmt>` rate
- `<goal-rate-decl>` rate

AST fields followed: `Quantity`/`Rate` references in the six declaration
classes became `Expr`. `LetDecl` was lifted out of `RuleDecl` and given a
top-level home in `AthleteBlock.lets`. The literal forms (`80 kg`,
`2.5 kg/week`) still parse — `parseAtom` already returns `QuantityExpr` or
`RateExpr` for them — so every existing sample continued to parse
unchanged after I moved its lets to the prefix.

Type checker grew per-slot helpers that call `checkExpr` and verify the
result is the family-shaped `FType` the slot requires (e.g. exercise weight
must be `TQuantity(MASS)`, progress rate must be `TRate(MASS, TIME)`). The
prefix lets are processed before workouts/meals, so their bindings are in
the env when the slot expressions are checked.

Interpreter mirrors the order: prefix lets feed `env` first, then workouts
and meals. Two thin downcast helpers (`evalMass`, `evalRateMassTime`) wrap
`evalExpr` at the slot call sites; the type checker has already proved the
shape, so the casts are total.

`valid_05_expressions.fl` is the capability sample: lets used in exercise
weight (`benchBase + 2 * plateStep`), routine args (none here — see below),
macro values (`2 * proteinPerMeal`), progress rate (`progRate`), and goal
rate (`weeklyGain`). The 4-week run shows Bench Press 85 → 87.5 → 90 kg
with bw projection 80 → 80.5 → 81 kg.

## Accepted

- **Lets are prefix-only.** Removed `LetDecl` from `RuleDecl`. Two of the
  existing samples (`valid_03_diet_only.fl`, `invalid_typeerror.fl`) moved
  their lets up. Single mental model — there is exactly one place a `let`
  may appear. Simpler than allowing both positions.
- **Slot helpers in the interpreter are downcasts, not re-checks.** `evalMass`
  is literally `(MassV) evalExpr(e)` — if the cast ever fails it is a type
  checker bug, not user input. Matches the philosophy that survived entry
  11: the checker proves, the interpreter computes.
- **No new operator rules.** The type system already admitted Number
  scaling and Rate × Time multiplication; the §4.7 widening just lets those
  operators reach more positions. Did NOT add `Quantity / Time → Rate` (see
  Errors caught below) — would have required a new §4.5 rule for one
  syntactic convenience.

## Rejected / modified

- **First attempt at the capability sample used `plateStep / 1 week` as a
  rate expression.** The type checker rejected it: `Mass / Time` is not in
  §4.5.4–§4.5.5. Replaced with a let-bound rate literal (`let progRate =
  2.5 kg/week`). The right fix — and it confirms the language design's
  stance that `Rate` is a separate kind, not derivable by syntactic
  division.
- **Did not allow lets in two positions.** The grammar could have admitted
  `<let-decl>` both as a prefix and as a `<rule-decl>` alternative; the
  interpreter and type checker already handled the rules-section case from
  P1. Rejected because "where can I write a let?" should have one answer,
  not two. Matches the keep-it-simple guidance.
- **Did not change the `parseAtom` rate-disambiguation.** The same
  `NUM_LIT UNIT "/" UNIT` look-ahead now serves slot positions transparently
  — no new disambiguation logic needed because the expression parser is the
  same as the let-RHS parser from P1.

## Errors I caught

- **§4.6.3 vs §4.6.5 scoping tension.** Caught before writing code by
  re-reading both sections. If I had skipped this check the natural
  one-pass design would have failed for any sample that referenced a
  let-bound name in a workout body — the let wouldn't be in the env yet.
  Asked the user to choose the resolution rather than guessing.
- **`MacroDecl.quantity` → `value`.** Renamed the field while widening the
  type. Justified: "quantity" was the old type's name, "value" matches the
  new role (any Quantity(MASS) expression). Caught the downstream sites
  (`TypeChecker.checkMealDecl`, `Interpreter.evalMeal`,
  `AstDumper.dumpMealDecl`) by compile errors after the AST edit.
- **`TargetDecl.thresholdEcho` displays canonical g, not declared unit.**
  The P1 `echoQuantity(t.qty)` printed the user's literal (`1.8 g`). Now
  that `qty` is an expression, there is no "declared unit" to echo — only
  the evaluated canonical-g value. Accepted the loss of source-unit
  fidelity in the target line because the alternative (synthesising a
  display unit from the expression tree) is significantly more work for a
  display-only gain.
- **Slot helpers vs `liftQuantity`.** First sketch kept the old
  `liftQuantity(ast.Quantity)` calls. That couldn't compile after the AST
  widening — the fields no longer held `Quantity`. Fixed by introducing the
  expr-based helpers; the static type system told me exactly where to
  change.

## Reflection (2–3 sentences)

First I was worrying about let'sections. The capability sample's first
draft failing on `Mass / Time` was a useful reminder that widening surface
forms doesn't widen the type algebra: the §4.5 rules are the actual
language, and §4.7 is just where they can reach.
