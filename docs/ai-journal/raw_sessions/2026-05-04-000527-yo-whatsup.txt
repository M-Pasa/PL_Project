● Before I draft §4.2, there are a few lexical-level design calls that
  materially shape the rest of the spec. I'd rather get your steer than pick
  silently.

  1. Open vs closed lexicon. Looking at the §4.1 sample, every unquoted token is
   either a reserved word (athlete, workout, weight, bulk), a unit (kg, g), or
  punctuation — every user-supplied name ("Push Day", "Bench Press") is a string
   literal. FitLang has no let / var and no user-named variables. I'd lean into
  that: closed lexicon, every unquoted lexeme reserved. It's defensible (cite
  Pascal's reserved-word philosophy, Sebesta §3) and makes the lexer trivially
  regular. Alternative: open lexicon with general identifiers, in case you want
  let restDay = ... later. Which?

  2. Unit suffixes — one token or two? 80 kg could be:
  - (A) a single QUANTITY_LIT lexeme via maximal-munch (Frink-style; units truly
   first-class, but lexer needs context for kg/week)
  - (B) two tokens NUM_LIT + UNIT paired at parse time (simpler regular lexer;
  closer to how C++ user-defined literals work pre-parse)

  I'd pick B — keeps the lexer regular, and rate forms (kg/week, g per kg)
  compose naturally in the grammar. But A is more "units are part of the
  lexicon."

  3. Comments. # to-end-of-line (config-file feel, matches the DSL's vibe) or //
   + /* */ (Java-family, since the interpreter is Java)? I'd go # only.

  default, Sebesta §3 calls Fortran's case-insensitivity an outlier). UTF-8
  source; identifiers ASCII; string literals allow Unicode so meal names like
  "Kahvaltı" work.

  5. Numeric literals. Integers and floats only (80, 2.5, 0.5) — no
  hex/octal/exponent (zero domain need). Negatives via unary -, not part of the
  literal.

● Good call holding on #1 — it's the one with downstream consequences. Let me
  lay it out concretely.

  What "closed lexicon" means in practice

  Every token the lexer can produce falls into one of these classes, and the 
  user cannot introduce a new word that the scanner doesn't already know:

  Class: Reserved words
  Examples: athlete, workout, exercise, meal, progress, target, goal, plan,
  week,
    with, sets, reps, weight, bodyweight, protein, carbs, fat, per, of
  Source: fixed in the language
  ────────────────────────────────────────
  Class: Mode literals
  Examples: bulk, cut, maintain, lose, gain
  Source: fixed (enum-like)
  ────────────────────────────────────────
  Class: Units
  Examples: kg, lb, g, kcal, kJ, s, min, h, day, week
  Source: fixed
  ────────────────────────────────────────
  Class: Numeric literals
  Examples: 80, 2.5
  Source: user-supplied values
  ────────────────────────────────────────
  Class: String literals
  Examples: "Push Day", "Bench Press", "Breakfast"
  Source: the only way the user introduces names
  ────────────────────────────────────────
  Class: Punctuation
  Examples: { } [ ] ( ) , : / >= <= > < =
  Source: fixed

  So when the user writes workout "Push Day" { ... }, the name "Push Day" is
  data — a string key in a workout registry. The grammar is closed; the data is
  open. Same idea as SQL (every unquoted token is a keyword; table/column names
  are unquoted identifiers, but that's because SQL has identifiers — strip those
   and you get FitLang) or JSON (closed grammar, all user names are quoted
  strings).

  What "open lexicon" would buy you

  Adding general identifiers means lines like:
  let pushDay = workout { ... }
  let dailyKcal = 2800 kcal
  plan week with workouts: [pushDay]

  You'd get named bindings — reuse a workout in multiple weeks, parameterize a
  meal by a calorie target, etc. That's a genuine feature, not just a syntactic
  preference. It also gives §4.6 (Names, Binding, Scope, Lifetime) much more to
  talk about — with a closed lexicon, §4.6 becomes "FitLang has no
  user-introduced names; the binding story is entirely about reserved-word-keyed
   slots in lexically nested blocks," which is a defensible but thin §4.6.

  The trade-off

  Column 1: Lexer
  Closed lexicon: Trivially regular, no symbol table at scan
  Open lexicon (let-bindings): Same (still regular)
  ────────────────────────────────────────
  Column 1: Parser
  Closed lexicon: Simpler — every keyword is a literal terminal
  Open lexicon (let-bindings): Slightly more — IDENT token, ambiguity between
    IDENT and reserved word
  ────────────────────────────────────────
  Column 1: §4.6 content
  Closed lexicon: Thin — only string-keyed registries (workout/meal name →
  block)
  Open lexicon (let-bindings): Rich — scope rules, shadowing, lifetime of let
    bindings
  ────────────────────────────────────────
  Column 1: §4.5 type system
  Closed lexicon: Simpler — every block schema is fixed
  Open lexicon (let-bindings): Type-checking variable references, possibly
    let-polymorphism
  ────────────────────────────────────────
  Column 1: Originality / DSL identity
  Closed lexicon: Stronger — feels like a configuration language for training,
    not a scripting language
  Open lexicon (let-bindings): Weaker — drifts toward "yet another scripting DSL

    with units"
  ────────────────────────────────────────
  Column 1: Expressive power
  Closed lexicon: Can't reuse a workout under two weeks without copy-paste,
  can't
    extract a magic number
  Open lexicon (let-bindings): Can name and reuse anything
  ────────────────────────────────────────
  Column 1: Sebesta defense
  Closed lexicon: Cite Pascal's reserved-word strictness (§3) and SQL/JSON
  closed
    grammars; simplicity & orthogonality (§1.3.1)
  Open lexicon (let-bindings): Cite Algol/Pascal/Java; richer abstraction
  (§1.3.2
     writability via abstraction)

  My recommendation

  Closed lexicon for D1, with one explicit escape hatch: numeric let-bindings 
  only.

  That is — the user can write let bodyweightTarget = 80 kg and refer to it
  later, but cannot bind workouts/meals to identifiers. Reasons:

  - §4.6 still has real content (numeric let-bindings, lexical scope,
  immutability — let not var, since you said "no globals" in your design
  constraints).
  - Workouts/meals stay first-class blocks keyed by string names — DSL identity
  preserved.
  - The grading rubric for "Use of Sebesta Framework" rewards named trade-offs;
  this lets you write a clean §4.8 paragraph: "we admit user identifiers only at
   the leaf-value level because giving block-typed values names would push
  FitLang from a configuration DSL toward a scripting language."
  - Workload-wise: tiny additional lexer/parser work, and a much stronger §4.5
  type-system story (you get to talk about typing variable references against
  the unit system).

  Three shapes to choose from:

  - (i) Fully closed — no identifiers anywhere. Cleanest DSL identity, thinnest
  §4.6.
  - (ii) Closed + numeric let (my pick) — one identifier class, leaf-only.
  - (iii) Fully open — identifiers can name any value including blocks. Most
  expressive, weakest DSL identity.

  Which shape?

❯ if we chose ii today can we expand the language to iii later?                 

● Yes — (ii) → (iii) is a strictly additive extension, not a breaking one.
  Concretely:

  Lexer. Identical. (ii) already needs an IDENT token class with the regex
  [a-zA-Z_][a-zA-Z0-9_]* minus reserved words. (iii) uses the same token;
  nothing changes at the scanner level.

  Grammar. (ii) restricts where IDENT can appear — only on the LHS of let, on
  the RHS of leaf-value positions (where a QUANTITY_LIT or NUM_LIT is expected).
   (iii) loosens that to any value position, including block-typed positions
  like workouts: [pushDay]. That's a relaxation of a production, not a
  redefinition — every (ii) program is still a valid (iii) program.

  Type system. (ii) only has to type-check identifier references against
  scalar/quantity types. (iii) extends the type system to block-typed values
  (workout-typed, meal-typed). The type rules for scalars don't change; you're
  adding new ones. Again additive.

  Semantics. (ii)'s let x = 80 kg is sugar for substituting a value. (iii) has
  to decide whether let push = workout {...} is a value binding (the workout
  block is a first-class value, structurally) or a reference. That decision is
  the only real design call you'd defer — but you don't have to make it now to
  keep (iii) reachable; you only have to make it when you actually do (iii).

  The one thing that would lock you out of (iii): if in (ii) you make IDENT
  legal only in let-RHS scalar positions and you parse it via a separate
  non-terminal (scalar_expr) that's disjoint from value_expr, you'd later need
  to merge them. That's a 10-line grammar refactor, not a redesign. To make it
  even cheaper: define one value_expr non-terminal in §4.3 now, and in §4.5
  type-restrict identifier references to scalars. Then (iii) is purely a
  type-system relaxation in the Part 2 spec — zero grammar change.

  Sebesta angle for §4.8. This is exactly the kind of staged evolution the book
  treats well: cite §1.4 (language design influences) on how Algol-family
  languages added first-class procedures after first-class values were already
  in the grammar, rather than re-architecting. You can defend (ii) for D1 as
  "the minimum identifier surface needed to make §4.6 substantive," and flag
  (iii) in §4.8's rationale paragraph as a future relaxation.

  So: pick (ii) today with the grammar shaped to make (iii) a type-system-only
  change later. Want me to go with that?
