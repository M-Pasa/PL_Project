● Good — both points sharpen the picture. Let me lay it out:

  Point 1 — meal ≠ exercise. Agreed and forced by the grammar: meal and exercise
   are different reserved words, so the parser already emits two distinct AST
  node kinds. No type rule has to "decide" they're different — they never reach
  the same equivalence test.

  Point 2 — within a kind (two exercises, or two meals). Here's the thing: §4.3
  fixes one schema per kind. Every <exercise-decl> has exactly sets, reps, 
  weight in that order; every <meal-decl> has only protein, carbs, fat in any
  order. The grammar doesn't admit an exercise with extra or missing fields. So:

  - Name equivalence answer: two exercises are the same type because both are
  "an exercise."
  - Structural equivalence answer: two exercises are the same type because their
   fields match 1-to-1.
  - In FitLang these collapse to the same answer — because the grammar
  guarantees every exercise has the same structure, name ⇔ structural for
  built-in records.

  The meaningful difference only appears if users can define their own schemas
  (e.g. an exercise2 { sets, reps, weight, tempo }). FitLang has no such syntax,
   so the distinction is hypothetical.

  Recommendation: call it name equivalence, because (a) the kind keyword
  (exercise / meal) is what carries identity, (b) it's the rule you can defend
  in one sentence in the quiz — "two values are the same type iff they were
  built from the same declared schema kind" — and (c) it matches how the parser
  actually tags nodes. The E2 experiment then has a clean comparison: AI will
  likely also suggest structural; you note they coincide here, and explain why
  you picked name (kind keyword is load-bearing; future schema extensions would
  need a new keyword, not a structural reshape).

  Want to lock that in and move to drafting §4.5, or push on it more?

● Good catch — let is in the P1 grammar but the interpreter/checker never
  honored it, so P2 has to decide what it actually binds. Three plausible
  scopes:

  Option A — Scalars + quantities only (recommended).
  let workingWeight = 80 kg, let dailyKcal = 2800 kcal, let proteinPerKg = 1.8 
  g. Reference them anywhere an <expression> is admitted — inside exercise { 
  weight: workingWeight }, inside target protein >= proteinPerKg per kg of 
  bodyweight. This is Sebesta §5.8 named constants (Pascal const, Ada constant,
  C #define/const, Rust const).

  Why this fits FitLang: the actual reuse pain in a real training program is "I
  want my top-set weight to appear in three exercises and stay in sync if I bump
   it." That's a scalar/quantity problem. Naming a number once is exactly what
  let is for.

  Option B — Also let-bind exercises and meals.
  let bench = exercise "..." { ... }, then put bench inside a workout body.
  Don't do this. It duplicates two existing mechanisms:
  - routine "Bench Triple" (load: kg) { exercise ... } already covers
  parameterized exercise reuse.
  - workout "Push Day" { ... } already names a workout via its STRING_LIT label.

  You'd end up with two name spaces fighting ("Bench Press" the label vs bench
  the IDENT) and an awkward question of how a let-bound exercise enters a
  workout. Plus it breaks the §4.6.1 "labels in the string space, scalars in the
   IDENT space" cleanliness.

  Option C — Also bind rates (let bulkRate = 0.25 kg/week).
  Cheap extension of A — same type rules, just adds <rate> to the RHS. Useful
  for goal gain bulkRate and progress weight bulkRate. I'd fold this into A.

  Recommendation: A + C. let binds scalars, quantities, and rates. One sentence
  to defend: "let introduces named constants for numeric values (Sebesta §5.8); 
  everything that has a label in the domain already gets one in the string-label
   space."

  Sound right? If yes, I'll fold it into §4.5 along with the name-equivalence
  call and start drafting.

● §4.4 drafted. Design calls baked in — flag any you want to change:

  1. Program output is one WeeklyPlan record (printed as text). No multi-week
  projection — progress just attaches nextWeekWeight per exercise. A multi-week
  extension would change only the projection horizon.

❯ should we allow long term programs or weekly plans are enough?                

● Good push. Three honest options:

  A — Keep single-week (what's drafted).
  plan week ... produces one week. progress shows up only as a nextWeekWeight
  annotation per exercise. goal lose/gain X kg/week is reported but not
  projected.
  - Pro: smallest semantics, smallest D3 output, fastest to D2 in the 6 days
  left.
  - Con: progress and goal lose/gain barely do work — they exist mostly to
  demonstrate Rate × Time without using it.

  B — Multi-week with a horizon (recommended).
  plan N week with ... where N is a NUM_LIT (default 1 if user writes just plan 
  week). Output becomes [WeeklyPlan_1, ..., WeeklyPlan_N]. In week k each
  exercise's weight is base + rate × (k−1 week); the projected bodyweight is
  bodyweight + goalRate × (k−1 week); target ... per kg of bodyweight recomputes
   per week.
  - Pro: progress, goal lose/gain, and the Rate × Time → Quantity rule become
  load-bearing — they're the only way to write a meaningful N-week program. The
  DSL identity gets stronger ("a FitLang program is a typed training block, not
  a snapshot"). One-sentence quiz defense: "periodization is the unit of 
  planning in strength training; a single week is the degenerate case."
  - Cost: one grammar line (<schedule-stmt> → "plan" [ NUM_LIT ] "week" "with" 
  ...), one for k in 1..N loop in the interpreter, slightly longer D3 outputs.
  Maybe 30 minutes of extra interpreter code.

  C — Block of named weeks (overkill).
  block "Hypertrophy" { week 1 { ... } week 2 { ... } ... } — explicit per-week
  declarations. Don't do this; it duplicates progress and removes the reason
  progress exists.

  My recommendation: B. It costs almost nothing and earns back the keywords. The
   grammar change is minimal:

  <schedule-stmt>    → "plan" [ NUM_LIT ] "week" "with" <schedule-clauses>

  (NUM_LIT here is a positive integer; absence means 1.) Type rule: nothing
  changes — the projection happens in the interpreter, the type checker still
  sees one schedule.

      738  ## 4.8 Design Rationale **[P2]**
      739  
      740 +This section restates the major design decisions as decisions 
          +— what was on the table, what was picked, and what was given u
          +p to pick it. Each paragraph names one trade-off Sebesta would
          + recognise.                                                   
      741 +                                                              
      742 +**Closed lexicon, no shadowing.** Every alphabetic lexeme is e
          +ither a reserved word, mode literal, unit, or an `IDENT` disjo
          +int from all three (§4.2.5, §4.6.3). We pay the rigidity Sebes
          +ta credits Pascal for (§3.1) — a user cannot use `weight` as a
          + let-name — and we earn the property that the parser never has
          + to disambiguate keyword from identifier and the type checker 
          +never has to resolve a shadowed binding. For a DSL whose users
          + are not software engineers, the rigidity cost is invisible (n
          +o one wants to call their let-binding `weight`) and the predic
          +tability payoff is real (every error message can point at a si
          +ngle declaration site).                                       
      743 +                                                              
      744 +**Strong typing with dimensional families.** §4.5.2 commits to
          + compile-time rejection of every cross-family operation and di
          +sallows same-family `*` and `/` outside the named `per UNIT of
          + bodyweight` sugar. The cost is the ergonomic ceremony Sebesta
          + credits Java for paying against C in §1.6 — `80 kg + 200` is 
          +rejected, the user writes `+ 200 g` (or `+ 0.2 kg`). The payof
          +f is that the entire class of "I mixed kcal and kJ" or "I wrot
          +e a kg where I meant g" bugs is eliminated at parse-plus-check
          + time, before the lifter sees a single nonsensical training we
          +ek.                                                           
      745 +                                                              
      746 +**Name equivalence on record kinds.** §4.5.3 picks name over s
          +tructural equivalence. The cost is the loss of structural flex
          +ibility — a hypothetical future kind that happens to share `ex
          +ercise`'s schema would not interoperate with it. The payoff is
          + the one-sentence quiz defence ("same kind ⇒ same type") and a
          + checker that needs no field-by-field comparison routine. Beca
          +use the grammar pins each kind to one schema, name and structu
          +ral agree on every legal program — the choice is *which rule t
          +o teach*, and name is the simpler teach.                      
      747 +                                                              
      748 +**Single-assignment, no mutation.** §4.6.6 commits to immutabi
          +lity. The cost is the inability to express imperative iteratio
          +n — `weight = weight + 2.5 kg` is not writable. The payoff is 
          +that lifetime = scope (§4.6.4), no dangling references, no ini
          +tialise-before-use rule, no garbage collection story, and a si
          +ngle-pass type checker. The iterative content the domain needs
          + (week-over-week weight) is recovered declaratively by `progre
          +ss` + `plan N week`, which is honest to how a lifter actually 
          +writes a periodised block: a base, a rate, a horizon.         
      749 +                                                              
      750 +**Static (lexical) scoping with a flat athlete-scope env.** §4
          +.6.3 takes Sebesta §5.5.1's C/Pascal/Java lineage and rejects 
          +dynamic scoping outright. The cost — none, given no first-clas
          +s procedures — is genuinely zero in this language, and the pay
          +off is that the referencing environment at any program point i
          +s computable from the AST alone. Routines (§4.4.7) are scoped 
          +at their declaration site, not at the call site, for the same 
          +reason: the lifter should never write a routine whose meaning 
          +depends on which workout it is called from.                   
      751 +                                                              
      752 +**Multi-week schedule as a first-class horizon.** §4.4.8 makes
