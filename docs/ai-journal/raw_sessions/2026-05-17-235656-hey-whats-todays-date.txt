‚ĚĮ what about when-decl scoping I didn't get that?                               

‚óŹ The question is: if you write a let inside a when branch, can later rules 
  (outside the when) see it?

  Concretely, this program:

  when goal: bulk {
      let surplusKcal = 300 kcal
      target protein >= 2.0 g per kg of bodyweight
  } else {
      target protein >= 1.6 g per kg of bodyweight
  }

  # Can a target down here reference `surplusKcal`?
  target carbs >= surplusKcal   # ‚Üź is this legal or unbound?

  Two defensible readings of ¬ß4.4.10:

  1. Branch-local (what I implemented). surplusKcal is scoped to the then-branch
   only. After the when-decl ends, the env is restored. Code outside the when
  sees the env as it was before. Result: surplusKcal is unbound in the trailing
  target carbs. Type error.
  2. Splice-and-leak (the literal reading of ¬ß4.4.10). The spec says: "the 
  surviving rule list is spliced into the enclosing <rules-section>." If you
  take that literally, then after static dispatch the program is equivalent to:

  let surplusKcal = 300 kcal      # spliced from then-branch
  target protein >= 2.0 g per kg of bodyweight
  target carbs >= surplusKcal      # ‚Üź now legal

  2. So surplusKcal does leak out, but only along the surviving branch ‚ÄĒ at
  compile time, the checker has already picked one branch via the goal mode.

  Why this matters at all: ¬ß4.4.10 says when is static ‚ÄĒ the dispatch value
  (goal mode) is a literal in the header, known at check time. So technically
  the checker could pick the surviving branch and check only that one, with
  bindings leaking into the trailing rules. That's reading (2).

  I went with reading (1) because:
  - It type-checks both branches (catches bugs in the dead branch ‚ÄĒ a future
  header change wouldn't suddenly expose a broken else).
  - Java treats if (false) { ... } the same way (still type-checked).
  - Conservative ‚ÄĒ never silently exposes a name from a branch that didn't run.

  But reading (2) is closer to what ¬ß4.4.10's "splice" wording promises, and is
  consistent with ¬ß4.4.10 calling this "the operational use of compile-time 
  dispatch."

  Quiz risk: if the grader points at ¬ß4.4.10's "splice" sentence and asks "so a
  let inside the surviving branch is visible after the when?", reading (1) gives
   the answer "no, branch-local" ‚ÄĒ which contradicts the literal spec. Reading
  (2) matches the spec but means the checker has to commit to one branch.

  Which one do you want? My recommendation, given the user-defendability
  priority: match the spec. Pick the surviving branch via the header goal, check
   only that branch, let its bindings leak. Update ¬ß4.4.10 if needed (it's
  already consistent with this reading) and the checker becomes a few lines
  simpler too.

‚ĚĮ so matching the spec, would make the let bound outside the when branch if the

‚óŹ Exactly. Spec-matching behavior:

  - At check time, look at the athlete header goal.
  - Pick the surviving branch (then if goal matches the when mode, else
  otherwise ‚ÄĒ or nothing if no else).
  - Check only that branch.
  - Its lets extend the env going forward; later rules can reference them.
  - The dead branch is never checked.

  So:

  athlete(bodyweight: 78 kg, goal: bulk) {
      ...

      target carbs >= deficitKcal g     # ‚Üź UNBOUND (else-branch dead, never
  checked)
  }

  And if the header were goal: cut, the opposite ‚ÄĒ deficitKcal would be bound,
  surplusKcal would not exist at all.

  compile error under this reading. If you wrote let bad = 80 kg + 200 kcal
  inside the dead else, the program type-checks fine. Some would call that a
  feature (matches Sebesta ¬ß8.1 static-if behavior ‚ÄĒ the dead code is dead, full

  you should be ready to say which side you're on.
