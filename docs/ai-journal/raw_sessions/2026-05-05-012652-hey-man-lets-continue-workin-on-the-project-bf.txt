● Right — the question is what the start symbol of the grammar looks like, i.e.
  what a .fit file is at the top level. Two real-language shapes to pick from:

  Option A — Structured program (Pascal-style).
  The book itself uses this in Sebesta §3.2/4.3 examples. A program is a fixed
  skeleton with named sections in a required order:

  program     → "program" identifier goal_section plan_section log_section "end"
  goal_section→ "goal" "{" goal_decl_list "}"
  plan_section→ "plan" "{" workout_list "}"
  log_section → "log"  "{" meal_list "}"

  Used by: Pascal (program ... begin ... end.), Ada (procedure Main is ... begin
   ... end Main). Pro: the grammar itself enforces "every FitLang program has
  exactly one goal, one training plan, one intake log" — no semantic check
  needed. Con: rigid; hard to write a tiny 3-line example.

  Option B — Flat sequence of declarations (C/Python-style).
  Start symbol is just a list:

  program     → declaration { declaration }
  declaration → goal_decl | workout_decl | meal_decl | rule_decl

  Used by: C (translation-unit = list of external declarations), Python (file =
  list of statements). Pro: flexible, easy small examples, easy to extend (add a
   new decl kind = one new alternative). Con: "must have exactly one goal"
  becomes a semantic/type-system rule, not a grammar rule.

  My recommendation: A. FitLang's identity is the coherent goal+plan+log triple
  — baking that into the syntax makes the DSL's domain claim visible in the
  grammar itself, which directly defends the originality rubric. It also gives
  §4.5 (type system) less work later.

  Which one?

● Exactly — that's the core argument. A general-purpose language can't assume
  what a "program" is for, so it gives you a flat list and lets you build
  anything. A DSL knows: a FitLang program is "pursue one goal via this plan,
  tracked by this log." Encoding that triple in the start symbol makes the
  grammar itself express the domain commitment.

  It also pays off downstream:
  - §4.5 type system can assume a single goal context exists — progression rules
   and macro targets resolve against it without scope-lookup logic.
  - §4.8 rationale gets a concrete originality argument: "the grammar refuses to
   parse a plan without a goal, because a plan without a goal isn't a fitness
  program."
  - Error messages at parse time ("missing goal block") are sharper than
  deferred semantic errors.

  The tradeoff is honest: you lose the ability to write a 2-line toy example.
  But for a domain-specific submission that's the right tradeoff — Pascal made
  the same call for teaching structured programming, and Ada doubled down for
  safety-critical work.

● Correct — and it's a deliberate call, not an oversight. There are three uses
  of = and : in the grammar, none of which are assignment:

  ┌────────────┬─────────────────────────────┬───────────────────────────────┐
  │   Token    │      Grammar position       │             Role              │
  ├────────────┼─────────────────────────────┼───────────────────────────────┤
  │ =          │ <let-decl> → "let" IDENT    │ Binding — introduces a name,  │
  │            │ "=" <expression>            │ never rebinds                 │
  ├────────────┼─────────────────────────────┼───────────────────────────────┤
  │ :          │ bodyweight: 78 kg, sets: 4, │ Field initialisation inside a │
  │            │  protein: 40 g              │  record literal               │
  ├────────────┼─────────────────────────────┼───────────────────────────────┤
  │ >= <= ==   │ <rel-op> in <target-decl>   │ Relational comparison, not    │
  │ != > <     │                             │ assignment                    │
  └────────────┴─────────────────────────────┴───────────────────────────────┘

  There is no mutation in FitLang. let introduces an immutable binding (Haskell
  let, Erlang single-assignment, Rust let without mut, ML val). Once a workout,
  meal, or let-bound name is declared, its value is fixed for the lifetime of
  the program.

  Why this is the right call for the domain: a fitness program is a plan, not a
  process. You don't mutate "Push Day" mid-execution — if you change it, you've
  defined a different workout. Same for meals and targets. Mutation would buy
  nothing the domain asks for and would force §4.5 to grow a reference/aliasing
  story (Sebesta §6.4.4–§6.5: l-values vs r-values, sharing, aliasing) that pure
   declarative bindings sidestep entirely. Sebesta credits this same trade-off
  to functional languages in §5.4.3.3 and to single-assignment dataflow
  languages in §15.

  Consequence for §4.6: "lifetime" becomes a one-line discussion — every
  binding's lifetime equals the enclosing block's lifetime, because nothing can
  outlive its block when nothing can be reassigned to escape. No heap, no
  pointers, no dangling references. That's worth stating explicitly as a
  rationale point.

  one of its anchor claims, then write the rest of the section around it?
