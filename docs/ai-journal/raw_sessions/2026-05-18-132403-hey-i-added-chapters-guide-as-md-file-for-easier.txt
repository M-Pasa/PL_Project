в—Ҹ Here's the design sketch. Grounded in D1 В§4.4 (which already nails most of it
  down).

  Interpreter design вҖ” tree-walking, post-typecheck

  Pipeline. Main runs Lexer вҶ’ Parser вҶ’ TypeChecker вҶ’ Interpreter. The
  interpreter assumes a well-typed AST: no runtime type guards, no "operands
  must agree" checks. Type errors are the checker's job; the interpreter only
  computes.

  What it produces. Per В§4.4.2, the denotation of a program is a single Plan
  value:
  Plan { athlete, N, weeks: [Week_1 вҖҰ Week_N] }
  Week { index, bodyweight, workouts: [WorkoutResult], targets: [TargetResult] }
  Output is Plan.toString() (a human-readable table). No side effects, no I/O,
  one-shot вҖ” same purity stance as the type system.

  Runtime values (Value sealed hierarchy):
  - NumV(double) вҖ” scalar
  - MassV(double grams), EnergyV(double kcal), TimeV(double seconds) вҖ” canonical
   units only
  - RateV(double, numerFamily, denomFamily) вҖ” rate, canonical/canonical
  - MacroV(MacroName, MassV) вҖ” for meal contents

  Per В§4.4.3, canonicalize at construction: a QuantityExpr(78, "kg") becomes
  MassV(78000). Surface units are remembered only at print time (interpreter
  picks a reasonable display unit, or we record the source unit on each leaf if
  we want exact echo вҖ” TBD, lean toward "always print in canonical + source").

  Environment. A simple Map<String, Value> chain вҖ” Env { parent, bindings }. Two
   levels are enough for P1/P2:
  1. Athlete env вҖ” built once from <let-decl> RHS evaluation (В§4.4.5). All lets
  are top-level inside the athlete block.
  2. Routine call env вҖ” extends athlete env with parameter bindings when a
  RoutineCall is evaluated (В§4.4.7).

  No closures, no mutation. Single-assignment everywhere (В§4.6.6).

  Expression evaluator. Straight recursive eval(Expr, Env) вҶ’ Value:
  - NumExpr вҶ’ NumV
  - QuantityExpr вҶ’ canonicalize вҶ’ MassV/EnergyV/TimeV
  - RateExpr вҶ’ RateV
  - IdentExpr вҶ’ env lookup (checker guarantees defined)
  - UnaryExpr("-", e) вҶ’ negate value
  - BinaryExpr вҖ” dispatch on operand kinds as guaranteed by the checker
  (В§4.5.4вҖ“5): Mass+Mass, MassвҲ’Mass, NumВ·Mass, MassВ·Num, Mass/Num, RateВ·Time,
  TimeВ·Rate. No general "type-tag-and-switch" вҖ” a small dispatch table keyed by
  (op, leftClass, rightClass).

  Declarations. Workouts and meals don't get "evaluated" at declaration time вҖ”
  they're stored as AST and only evaluated when the <schedule-stmt> names them
  (В§4.4.6). Routines likewise stored, invoked on RoutineCall (В§4.4.7).

  The plan N week loop (В§4.4.8) вҖ” the heart of execution:
  for k in 1..N:
      bw_k        = project bodyweight at week k using GoalRateDecl (Rate Г— Time
   = Mass)
      workouts_k  = [ evalWorkout(label, k) for label in schedule.workouts ]
      macros_k    = sum of meal macros (meals are constant across weeks in P2)
      targets_k   = [ evalTarget(t, macros_k, bw_k) for t in rules-in-scope ]
      weeks += Week(k, bw_k, workouts_k, targets_k)

  when evaluation (В§4.4.10) вҖ” dispatch on athlete.params.goal once, flatten the
  chosen branch's rules into the rules-in-scope list. No per-week re-evaluation
  since the mode is fixed for the program.

  Exercise weight per week. In P1 weights are literal <quantity> (no expressions
   allowed inside workout bodies вҖ” В§4.6.5). For P2, if В§4.7 has been extended to
   allow expressions there, evaluate under athlete env + an implicit week
  binding. If not yet extended, just echo the literal.

  Target checks (В§4.4.9). For each TargetDecl, evaluate the threshold (which may
   be q per bodyweight, i.e. qty * bw_k after dimensional cancellation), compare
   against the planned macro total, emit OK or OFF by X.

  Error policy. Only one runtime error class survives type-checking:
  divide-by-zero in a Num/Num expression. Throw FitLangException with line info.
   Everything else is a type/scope error caught earlier.
