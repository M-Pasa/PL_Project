## 1. Valid programs

### 1.1 `valid_01_full.fl` — full bulk-phase program

Demonstrates the full §4.3 skeleton: `<plan-section>` with a `<progress-stmt>`, `<intake-section>`, three `<rule-decl>`s including a `<when-decl>` (the §2 control structure) with both branches, and the mandatory `<schedule-stmt>` closing the block.

**Source:**

```fitlang
# Full FitLang example — bulk phase with workout + nutrition + rules
athlete(bodyweight: 78 kg, goal: bulk) {

    workout "Push Day" {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: 80 kg }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: 50 kg }
        progress weight 2.5 kg/week
    }

    meal "Breakfast" {
        protein: 40 g
        carbs:   80 g
        fat:     15 g
    }

    target protein >= 1.8 g per kg of bodyweight
    goal   lose 0.5 kg/week

    when goal: bulk {
        target protein >= 2.0 g per kg of bodyweight
    } else {
        target protein >= 1.6 g per kg of bodyweight
    }

    plan week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```

**`--dump-ast` output:**

```
AthleteBlock
  AthleteParams
    bodyweight : 78.0 kg  [MASS]
    goal       : bulk
  WorkoutDecl "Push Day"
    ExerciseDecl "Bench Press"
      sets   : 4
      reps   : 6
      weight : 80.0 kg  [MASS]
    ExerciseDecl "Overhead Press"
      sets   : 3
      reps   : 8
      weight : 50.0 kg  [MASS]
    ProgressStmt rate=2.5 kg/week
  MealDecl "Breakfast"
    MacroDecl protein : 40.0 g  [MASS]
    MacroDecl carbs : 80.0 g  [MASS]
    MacroDecl fat : 15.0 g  [MASS]
  RulesSection
    TargetDecl protein >= 1.8 g per kg of bodyweight
    GoalRateDecl lose 0.5 kg/week
    WhenDecl goal=bulk
      then
        TargetDecl protein >= 2.0 g per kg of bodyweight
      else
        TargetDecl protein >= 1.6 g per kg of bodyweight
  ScheduleStmt
    workouts : ["Push Day"]
    meals    : ["Breakfast"]
```

**Constructs exercised.** `<athlete-params>` (mass quantity + goal mode), `<plan-section>` with `<progress-stmt>` (3-token rate `2.5 kg/week`), `<intake-section>` with `<macro-decl>` repetition, all four `<rule-decl>` alternatives including the new `<when-decl>` with `else`-branch, mandatory `<schedule-stmt>` with both `workouts:` and `meals:` clauses.

### 1.2 `valid_02_workout_only.fl` — routine template + workout-only body

Demonstrates the §2 user-defined function template: a parameterised `<routine-decl>` declared once at athlete scope and invoked from a `<workout-decl>` body via `<routine-call>` (`use "Base Press"(82 kg, 22 kg)`). Also shows that the `<intake-section>` is optional — workout-only programs are first-class.

**Source:**

```fitlang
# Workout-only program — no nutrition section (diet-only and workout-only are both legal)
# Demonstrates routine declaration (parameterized template) and routine call.
athlete(bodyweight: 90 kg, goal: cut) {

    routine "Base Press"(load: kg, accessory_load: kg) {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: 80 kg }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: 50 kg }
    }

    workout "Pull Day" {
        exercise "Deadlift"   { sets: 3, reps: 5, weight: 140 kg }
        exercise "Barbell Row" { sets: 4, reps: 8, weight: 70 kg }
        progress weight 2.5 kg/week
    }

    workout "Push Day" {
        use "Base Press"(82 kg, 22 kg)
        exercise "Triceps Pushdown" { sets: 3, reps: 12, weight: 30 kg }
    }

    workout "Leg Day" {
        exercise "Squat"      { sets: 4, reps: 5, weight: 100 kg }
        exercise "Leg Press"  { sets: 3, reps: 10, weight: 120 kg }
    }

    goal lose 0.75 kg/week

    plan week with workouts: ["Pull Day", "Push Day", "Leg Day"]
}
```

### 1.3 `valid_03_diet_only.fl` — diet-only with `let` and per-bodyweight targets

Demonstrates the symmetric case (no `<plan-section>`, only `<intake-section>`), the additive `<expression>` layer in `<let-decl>`, and two `<target-decl>`s using the per-bodyweight scaling form.

**Source:**

```fitlang
# Diet-only program — no workout section (a client tracking nutrition only)
athlete(bodyweight: 65 kg, goal: maintain) {

    meal "Breakfast" {
        protein: 30 g
        carbs:   60 g
        fat:     10 g
    }

    meal "Lunch" {
        protein: 50 g
        carbs:   90 g
        fat:     20 g
    }

    meal "Dinner" {
        protein: 45 g
        carbs:   70 g
        fat:     18 g
    }

    let dailyProtein = 30 g + 50 g + 45 g

    target protein >= 1.6 g per kg of bodyweight
    target fat <= 0.8 g per kg of bodyweight

    plan week with meals: ["Breakfast", "Lunch", "Dinner"]
}
```

**`--dump-ast` output:**

```
AthleteBlock
  AthleteParams
    bodyweight : 65.0 kg  [MASS]
    goal       : maintain
  MealDecl "Breakfast"
    MacroDecl protein : 30.0 g  [MASS]
    MacroDecl carbs : 60.0 g  [MASS]
    MacroDecl fat : 10.0 g  [MASS]
  MealDecl "Lunch"
    MacroDecl protein : 50.0 g  [MASS]
    MacroDecl carbs : 90.0 g  [MASS]
    MacroDecl fat : 20.0 g  [MASS]
  MealDecl "Dinner"
    MacroDecl protein : 45.0 g  [MASS]
    MacroDecl carbs : 70.0 g  [MASS]
    MacroDecl fat : 18.0 g  [MASS]

    LetDecl dailyProtein = ((30.0 g + 50.0 g) + 45.0 g)
    TargetDecl protein >= 1.6 g per kg of bodyweight
    TargetDecl fat <= 0.8 g per kg of bodyweight
  ScheduleStmt

```

**Constructs exercised.** `goal: maintain` (third element of the `<goal-mode>` enumeration), three `<meal-decl>`s with no `<plan-section>` (`<body-sections>` selecting the right alternative), `<let-decl>` with a 3-term left-associative additive `<expression>` (parenthesisation in the dump confirms `(a + b) + c`), the per-bodyweight scaling form on `<target-decl>` with both `>=` and `<=` `<rel-op>`s, `<schedule-stmt>` with `meals:` only.

## 2. Malformed programs

For each malformed sample, the parser exits with a non-zero status and prints a single-line diagnostic of the form `Parse error: <message> [line L, col C]`. The exact diagnostics below are reproduced verbatim from the parser's stderr output.

### 2.1 `malformed_01_empty_body.fl` — empty athlete body

**Source:**

```fitlang
# MALFORMED: athlete block has no workout or meal — grammar requires at least one
athlete(bodyweight: 78 kg, goal: bulk) {
    plan week with workouts: ["Push Day"]
}
```

**Diagnostic:**

```
Parse error: Expected 'workout' or 'meal' block inside athlete block [line 4, col 5]
```

**Production violated.** §4.3.2 `<body-sections> → <plan-section> [ <intake-section> ] | <intake-section>`. Both alternatives require at least one `<workout-decl>` or `<meal-decl>` (each section is `+`, not `*`); no derivation produces an empty body. The parser reports the violation at the first token after `{`, which is `plan` — outside the FIRST set of `<body-sections>`.

---

### 2.2 `malformed_02_empty_workout.fl` — workout with no exercises

**Source:**

```fitlang
# MALFORMED: workout block contains no exercises — grammar requires at least one exercise
athlete(bodyweight: 78 kg, goal: bulk) {
    workout "Empty Day" {
    }
    plan week with workouts: ["Empty Day"]
}
```

**Diagnostic:**

```
Parse error: Workout "Empty Day" must contain at least one exercise or routine call [line 5, col 5]
```

**Production violated.** §4.3.2 `<workout-decl> → "workout" STRING_LIT "{" <workout-item> { <workout-item> } [ <progress-stmt> ] "}"`. The body is one-or-more (`<workout-item> { <workout-item> }`), not zero-or-more. The parser checks the FIRST set `{ exercise, use }` of `<workout-item>` immediately after `{` and rejects when neither is present.

---

### 2.3 `malformed_03_missing_schedule.fl` — no `<schedule-stmt>`

**Source:**

```fitlang
# MALFORMED: no plan statement — grammar requires exactly one schedule-stmt to close the block
athlete(bodyweight: 78 kg, goal: bulk) {
    workout "Push Day" {
        exercise "Bench Press" { sets: 4, reps: 6, weight: 80 kg }
    }
    meal "Breakfast" {
        protein: 40 g
        carbs:   80 g
        fat:     15 g
    }
}
```

**Diagnostic:**

```
Parse error: Expected PLAN but got RBRACE ('}') [line 12, col 1]
```

**Production violated.** §4.3.2 `<athlete-block>` ends with the mandatory `<schedule-stmt>` before the closing `}`. After `<rules-section>` consumes zero rules (FIRST disjoint from `}`), the parser expects `plan` to begin `<schedule-stmt>` and finds `}` instead. This is one of the three §4.3.3 invariants the start symbol enforces statically.

---

### 2.4 `malformed_04_bad_goal_mode.fl` — invalid `<goal-mode>`

**Source:**

```fitlang
# MALFORMED: goal mode "shred" is not in the closed lexicon (bulk/cut/maintain only)
athlete(bodyweight: 78 kg, goal: shred) {
    workout "Push Day" {
        exercise "Bench Press" { sets: 4, reps: 6, weight: 80 kg }
    }
    plan week with workouts: ["Push Day"]
}
```

**Diagnostic:**

```
Parse error: Expected goal mode: bulk, cut, or maintain [line 3, col 34]
```

**Production violated.** §4.3.2 `<goal-mode> → "bulk" | "cut" | "maintain"`. The lexer (§4.2.5 closed-lexicon rule) emits `IDENT` for `shred` because it is not a reserved word, mode literal, or unit. The parser then fails the three-alternative match for `<goal-mode>` and reports the closed enumeration directly. This shows the §4.2.5 closed-lexicon design paying off: there is no quiet acceptance of an unknown mode at parse time, only a single deterministic error.

---

### 2.5 `malformed_05_unterminated_string.fl` — string literal with no closing quote

**Source:**

```fitlang
# MALFORMED: string literal is not closed before newline
athlete(bodyweight: 78 kg, goal: bulk) {
    workout "Push Day {
        exercise "Bench Press" { sets: 4, reps: 6, weight: 80 kg }
    }
    plan week with workouts: ["Push Day"]
}
```

**Diagnostic:**

```
Parse error: Unterminated string literal [line 4, col 13]
```

**Production violated.** §4.2.4 string-literal lexical rule: a `STRING_LIT` is `"` *any non-`"` non-newline characters* `"`. The lexer scans character-by-character and rejects the line break before the closing `"`. This is a lexical error rather than a syntactic one, but is reported through the same `Parse error:` channel because the parser owns the diagnostic surface. Detected at the first newline encountered inside the unterminated literal — line 4, column 13 — before any tokens past `workout` are emitted.
