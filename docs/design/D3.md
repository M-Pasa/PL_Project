## 1. Valid programs

### 1.1 `valid_01_full.fl` — full bulk-phase program

Demonstrates the full §4.3 skeleton: `<plan-section>` with a `<progress-stmt>`, `<intake-section>`, three `<rule-decl>`s including a `<when-decl>` (the §2 control structure) with both branches, and the mandatory `<schedule-stmt>` closing the block.

**Source:**

```fitlang
# Full FitLang example — bulk phase with workout + nutrition + rules
athlete(bodyweight: 78 kg, goal: bulk) {

    workout "Push Day" {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: 80 kg }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: 50 kg }
        progress weight 2.5 kg/week
    }

    meal "Breakfast" {
        protein: 40 g
        carbs:   80 g
        fat:     15 g
    }

    target protein >= 1.8 g per kg of bodyweight
    goal   gain 0.5 kg/week

    when goal: bulk {
        target protein >= 2.0 g per kg of bodyweight
    } else {
        target protein >= 1.6 g per kg of bodyweight
    }

    plan week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```


### 1.2 `valid_02_workout_only.fl` — routine template + workout-only body

Demonstrates the §2 user-defined function template: a parameterised `<routine-decl>` declared once at athlete scope and invoked from a `<workout-decl>` body via `<routine-call>` (`use "Base Press"(82 kg, 22 kg)`). Also shows that the `<intake-section>` is optional — workout-only programs are first-class.

**Source:**

```fitlang
# Workout-only program — no nutrition section (diet-only and workout-only are both legal)
# Demonstrates routine declaration (parameterized template) and routine call.
athlete(bodyweight: 90 kg, goal: cut) {

    routine "Base Press"(load: kg, accessory_load: kg) {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: load }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: accessory_load }
    }

    workout "Pull Day" {
        exercise "Deadlift"   { sets: 3, reps: 5, weight: 140 kg }
        exercise "Barbell Row" { sets: 4, reps: 8, weight: 70 kg }
        progress weight 2.5 kg/week
    }

    workout "Push Day" {
        use "Base Press"(82 kg, 22 kg)
        exercise "Triceps Pushdown" { sets: 3, reps: 12, weight: 30 kg }
    }

    workout "Leg Day" {
        exercise "Squat"      { sets: 4, reps: 5, weight: 100 kg }
        exercise "Leg Press"  { sets: 3, reps: 10, weight: 120 kg }
    }

    goal lose 0.75 kg/week

    plan week with workouts: ["Pull Day", "Push Day", "Leg Day"]
}
```

**Constructs exercised.** `<routine-decl>` with two `<routine-params>` (`IDENT ":" UNIT` pairs), each parameter referenced by name inside the routine body's `<exercise-decl>` `weight:` slot — so the routine is a real template parameterised over Mass, not a wrapper around literals. The call site `use "Base Press"(82 kg, 22 kg)` then substitutes `load = 82 kg` and `accessory_load = 22 kg` into the routine's two exercise weights (visible in `--run` as `Bench Press @ 82 kg` and `Overhead Press @ 22 kg`). Also exercised: `<routine-call>` mixed with `<exercise-decl>` inside a `<workout-decl>` body via `<workout-item>`, three `<workout-decl>`s in `<plan-section>` with no `<intake-section>` (right alternative of `<body-sections>` collapses to plan-only), `<goal-rate-decl>`, schedule with `workouts:` only. The substitution semantics matches §4.4.5: the call site evaluates each argument expression in the caller's environment, binds the values into the routine's parameter names, and evaluates the body in the extended environment — Pascal-style by-value parameter passing (Sebesta §9.5.1).

### 1.3 `valid_03_diet_only.fl` — diet-only with prefix `let` and per-bodyweight targets

Demonstrates the symmetric case (no `<plan-section>`, only `<intake-section>`), the additive `<expression>` layer inside the prefix `<let-section>` (§4.7 widening), and two `<target-decl>`s using the per-bodyweight scaling form.

**Source:**

```fitlang
# Diet-only program — no workout section (a client tracking nutrition only)
athlete(bodyweight: 65 kg, goal: maintain) {

    let dailyProtein = 30 g + 50 g + 45 g

    meal "Breakfast" {
        protein: 30 g
        carbs:   60 g
        fat:     10 g
    }

    meal "Lunch" {
        protein: 50 g

        protein: 45 g
```

### 1.4 `valid_04_multi_week.fl` — `plan N week` projection (§4.4.8)

### 1.5 `valid_05_expressions.fl` — prefix `let`s reach every value slot (§4.7)

Demonstrates the §4.7 expression-slot widening: a prefix `<let-section>` declares five names (two Mass quantities, two Mass/Time rates, one macro Mass quantity), each of which is referenced from one of the six value-position slots that P2 widened to `<expression>` — exercise weight, macro value, progress rate, goal rate, and inside arithmetic expressions in the slots themselves.

**Source:**

```fitlang
# §4.7: let bindings are now usable inside workout, meal, target, and rate slots.
athlete(bodyweight: 80 kg, goal: bulk) {

    let benchBase   = 80 kg
    let plateStep   = 2.5 kg
    let progRate    = 2.5 kg/week
    let weeklyGain  = 0.5 kg/week
    let proteinPerMeal = 40 g

    workout "Push Day" {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: benchBase + 2 * plateStep }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: benchBase - 30 kg }
        progress weight progRate
    }

    meal "Breakfast" {
        protein: proteinPerMeal
        carbs:   2 * proteinPerMeal
        fat:     proteinPerMeal / 2
    }

    target protein >= 1.8 g per kg of bodyweight
    goal   gain weeklyGain

    plan 3 week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```

**Constructs exercised.** All six §4.7-widened slots: (i) `<exercise-decl>` `weight:` uses the arithmetic expression `benchBase + 2 * plateStep` → `80 + 5 = 85 kg`, and `benchBase - 30 kg` mixes a let-bound `kg` value with a literal `kg` quantity; (ii) `<macro-decl>` value uses a bare identifier (`proteinPerMeal`), a `Number * Quantity` form (`2 * proteinPerMeal`), and a `Quantity / Number` form (`proteinPerMeal / 2`) — three of the four §4.5.4 multiplicative rules in one meal; (iii) `<progress-stmt>` rate is a bare identifier of `Rate(MASS,TIME)`; (iv) `<goal-rate-decl>` rate is also a bare `Rate(MASS,TIME)` identifier. The `--dump-ast` output preserves the unevaluated expressions (e.g. `weight : (benchBase + (2.0 * plateStep))`) — the interpreter resolves them only in the `--run` pass. ### 1.6 `valid_06_comprehensive.fl` — every D1 feature in one program

A single program designed as final-pass verification: it exercises every D1 feature jointly, so the constructs the earlier samples cover individually all appear in one well-typed program. Specifically: routine declaration with two typed parameters and a routine call that substitutes them, prefix `<let-section>` with ten bindings, within-MASS unit coercion (`80 kg + 200 g`), all four arithmetic operators in expression positions, unary minus, parenthesised grouping, identifier-typed rate in both `<progress-stmt>` and `<goal-rate-decl>`, the `Rate × Time → Quantity` rule (§4.5.5), `<when-decl>` with both arms, `<target-decl>` with both `<rel-op>`s, and `plan 3 week` multi-week unfolding with bodyweight projection and target re-evaluation.

**Source:**

```fitlang
# Comprehensive sample — exercises every D1 feature in one program.
# Covers: closed lexicon, athlete header, routine decl + call w/ substitution,
# prefix lets, within-family unit coercion (kg + g), all four arithmetic ops,
# unary minus, parens, identifier rates, Rate × Time → Quantity (§4.5.5),
# progress, goal-rate, target with per-bodyweight scaling (both rel-ops),
# when-decl with surviving and dead branches, plan N week multi-week unfolding,
# schedule with both workouts and meals.

athlete(bodyweight: 80 kg, goal: bulk) {

    routine "Base Press"(top: kg, accessory: kg) {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: top }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: accessory }
    }

    let base           = 80 kg + 200 g            # within-MASS coercion (g → kg)
    let plateStep      = 2.5 kg
    let topWeight      = base + 2 * plateStep     # binary + and Number * Quantity
    let accWeight      = -plateStep + 22.5 kg     # unary minus
    let halfBench      = (topWeight / 2)          # parens + Quantity / Number
    let progRate       = 2.5 kg/week              # rate literal
    let horizon        = 4 week                   # Quantity(TIME)
    let endBoost       = progRate * horizon       # Rate × Time → Mass (§4.5.5)
    let proteinPerMeal = 40 g
    let weeklyGain     = 0.5 kg/week

    workout "Push Day" {
        use "Base Press"(topWeight, accWeight)
        exercise "Triceps Pushdown" { sets: 3, reps: 12, weight: halfBench + endBoost - 30 kg }
        progress weight progRate
    }

    meal "Breakfast" {
        protein: proteinPerMeal
        carbs:   2 * proteinPerMeal
        fat:     proteinPerMeal / 2
    }

    target protein >= 1.8 g per kg of bodyweight
    target fat <= 1 g per kg of bodyweight
    goal gain weeklyGain

    when goal: bulk {
        target protein >= 2 g per kg of bodyweight
    } else {
        target protein >= 1.6 g per kg of bodyweight
    }

    plan 3 week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```

### 1.7 `valid_07_cut_branch.fl` — `when`'s `else` branch fires for `goal: cut`

## 2. Malformed programs

For each malformed sample, the parser exits with a non-zero status and prints a single-line diagnostic of the form `Parse error: <message> [line L, col C]`. The exact diagnostics below are reproduced verbatim from the parser's stderr output.

### 2.1 `malformed_01_empty_body.fl` — empty athlete body

**Source:**

```fitlang
# MALFORMED: athlete block has no workout or meal — grammar requires at least one
athlete(bodyweight: 78 kg, goal: bulk) {
    plan week with workouts: ["Push Day"]
}
```

**Diagnostic:**

```
Parse error: Expected 'workout' or 'meal' block inside athlete block [line 4, col 5]
```

**Production violated.** §4.3.2 `<body-sections> → <plan-section> [ <intake-section> ] | <intake-section>`. Both alternatives require at least one `<workout-decl>` or `<meal-decl>` (each section is `+`, not `*`); no derivation produces an empty body. The parser reports the violation at the first token after `{`, which is `plan` — outside the FIRST set of `<body-sections>`.

### 2.2 `malformed_02_empty_workout.fl` — workout with no exercises

**Source:**

```fitlang
# MALFORMED: workout block contains no exercises — grammar requires at least one exercise
athlete(bodyweight: 78 kg, goal: bulk) {
    workout "Empty Day" {
    }
    plan week with workouts: ["Empty Day"]
}
```

**Diagnostic:**

```
Parse error: Workout "Empty Day" must contain at least one exercise or routine call [line 5, col 5]
```

**Production violated.** §4.3.2 `<workout-decl> → "workout" STRING_LIT "{" <workout-item> { <workout-item> } [ <progress-stmt> ] "}"`. The body is one-or-more (`<workout-item> { <workout-item> }`), not zero-or-more. The parser checks the FIRST set `{ exercise, use }` of `<workout-item>` immediately after `{` and rejects when neither is present.

### 2.3 `malformed_03_missing_schedule.fl` — no `<schedule-stmt>`

**Source:**

```fitlang
# MALFORMED: no plan statement — grammar requires exactly one schedule-stmt to close the block
athlete(bodyweight: 78 kg, goal: bulk) {
    workout "Push Day" {
        exercise "Bench Press" { sets: 4, reps: 6, weight: 80 kg }

    meal "Breakfast" {
        protein: 40 g
        carbs:   80 g
        fat:     15 g
    }
```

**Diagnostic:**

```
Parse error: Expected PLAN but got RBRACE ('}') [line 12, col 1]
```

**Production violated.** §4.3.2 `<athlete-block>` ends with the mandatory `<schedule-stmt>` before the closing `}`. After `<rules-section>` consumes zero rules (FIRST disjoint from `}`), the parser expects `plan` to begin `<schedule-stmt>` and finds `}` instead. This is one of the three §4.3.3 invariants the start symbol enforces statically.

### 2.4 `malformed_04_bad_goal_mode.fl` — invalid `<goal-mode>`

**Source:**

## 3. Type-error programs

The five malformed programs above all fail at parse time — they violate §4.3 productions or §4.2 lexical rules. The complementary case is a program that *parses* (every token is well-formed and every production derives) but that the §4.5 / §4.6 type checker rejects. The interpreter is never reached: the checker collects every diagnostic in one pass and exits before `--run` evaluation. Two samples cover this path — one with ten diverse violations (§4.5 arithmetic, §4.5 record slots, §4.5 rate kind, §4.6 scope, §4.4 schedule), and one focused on the §4.6.3 no-shadowing rule.

### 3.1 `invalid_typeerror.fl` — ten distinct type errors in one program

The program below is deliberately ill-typed in ten different ways. Each numbered comment marks one violation the §4.5 / §4.6 rules forbid. The point is to verify that the checker catches *all* of them in one run (rather than stopping at the first), and that each diagnostic names the rule and the offending operand types.

**Source:**

```fitlang

# violation the §4.5 type checker is expected to report. Used for D3 evidence
# of type-checker behaviour and for E3's negative test inputs.
#

#   1. exercise weight is Energy, not Mass                   (§4.5.6 + §4.5.4)

#   3. meal macro is Energy, not Mass                        (§4.5.6)
#   4. cross-family addition: Mass + Energy                  (§4.5.4 rule 2)
#   5. same-family division: Mass / Mass (would yield Number) (§4.5.4 rule 2)
#   6. same-family multiplication: Mass * Mass (would yield Mass²) (§4.5.4 rule 2)
#   7. unbound identifier on let RHS                          (§4.6.3)
#   8. target "per UNIT of bodyweight" UNIT is Energy, not Mass (§4.4.9)
#   9. goal-rate has Mass denominator, not Time               (§4.5.5)
#  10. schedule references an undeclared meal label           (§4.4.8)
```

### 3.2 `invalid_shadow.fl` — `<let-decl>` shadowing rejected (§4.6.3)

A focused negative sample for the §4.6.3 no-shadowing rule: the same `<let-decl>` name is bound twice in the same athlete scope. The checker reports the violation at the second binding and does not attempt to type-check the rebinding's RHS.

**Source:**

```fitlang
# Type-error sample — no-shadowing rule (§4.6.3).
# The second `let benchBase` is illegal because the name is already bound
# at the same athlete scope. The checker is expected to emit a single
# diagnostic naming the rule.

athlete(bodyweight: 80 kg, goal: bulk) {

    let benchBase = 80 kg
    let plateStep = 2.5 kg
    let benchBase = 82 kg          # SHADOW — should be rejected

    workout "Push Day" {
        exercise "Bench Press" { sets: 4, reps: 6, weight: benchBase }
        progress weight 2.5 kg/week
    }

    plan week with workouts: ["Push Day"]
}
```

**Checker output (stderr):**

```
Type errors in 'samples\invalid_shadow.fl':
  - let "benchBase": shadowing is not allowed (§4.6.3)
```

Exit code `1`. The diagnostic names the rule directly; the rebinding does not silently overwrite the first binding (Algol-68 / Pascal style — Sebesta §5.5.4), and the following workout body still sees the *first* `benchBase = 80 kg` binding in the type checker's env (had we suppressed the error and continued, the exercise weight would type-check against 80 kg, not 82 kg). This is the §4.6.3 rule the closed-lexicon design depends on: every identifier in an athlete block has exactly one binding site.
