## 4.1 Language Overview **[P1]**

**Name.** FitLang.

**Domain.** A domain-specific language for writing personal strength-training programs together with the nutrition required to execute them. A FitLang program describes workout templates (sets, reps, loads), meals (macronutrients), and progression and nutritional rules — all evaluated against a single lexically scoped athlete context (bodyweight, goal). Physical quantities (`Mass`, `Energy`, `Time`) are first-class typed values; the language refuses to silently mix incompatible units.

**Sample program.**

```fitlang
athlete(bodyweight: 78 kg, goal: bulk) {
    workout "Push Day" {
        exercise "Bench Press" { sets: 4, reps: 6, weight: 80 kg }
        progress weight 2.5 kg/week
    }
    meal "Breakfast" { protein: 40 g, carbs: 80 g, fat: 15 g }
    target protein >= 1.8 g per kg of bodyweight
    goal gain 0.25 kg/week
    plan 4 week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```

**Justification (Sebesta §1.3).** FitLang optimises **writability** for its domain (§1.3.2 — writability is always relative to a domain): the surface syntax matches how a lifter writes a training log. **Reliability** is bought by a strict unit-aware type system (§1.3.3) that catches cross-family arithmetic at compile time — the same trade-off Sebesta credits Java for against C in §1.6, where mandatory checks cost execution speed but eliminate a class of bugs. Criteria knowingly sacrificed: **generality** (the language is useless outside fitness/nutrition — a secondary criterion per §1.3) and **cost of execution** (tree-walking interpretation is 10–100× slower than compiled code per §1.7, irrelevant for a sub-second plan calculation).

## 4.2 Lexical Structure **[P1]**

### 4.2.1 Character set and whitespace

Source is UTF-8; non-ASCII code points are permitted only inside string literals and line comments. Whitespace (space, tab, LF, CR) is insignificant. A `#` outside a string literal starts a line comment to end of line.

### 4.2.2 Token classes

| Class | Pattern (POSIX-extended regex) | Examples |
|---|---|---|
| **RESERVED** | closed table (§4.2.4) | `athlete`, `workout`, `let` |
| **MODE_LIT** | closed table (§4.2.4) | `bulk`, `cut`, `lose` |
| **UNIT** | closed table (§4.2.5) | `kg`, `kcal`, `week` |
| **NUM_LIT** | `[0-9]+(\.[0-9]+)?` | `80`, `2.5` |
| **STRING_LIT** | `"` *(Unicode except `"` and newline; `\\` `\"` escapes)* `"` | `"Push Day"` |
| **IDENT** | `[A-Za-z_][A-Za-z0-9_]*` minus closed lexicon | `workingWeight` |
| **PUNCT** | `{ } [ ] ( ) , : / + - * = > < >= <= == !=` | `{`, `>=` |

Numeric literals carry no sign; leading `-` is the unary minus operator. No integer/float distinction at the lexical level — both `80` and `80.0` produce a **NUM_LIT** token; admissibility is decided by the type checker (§4.5). Maximal-munch (Sebesta §4.2) resolves `>=`/`<=`/`==`/`!=` against their single-character prefixes.

### 4.2.3 Closed lexicon

Every unquoted alphabetic lexeme is a reserved word, mode literal, or unit — the "fully reserved" stance Sebesta credits to Pascal (§3.1). An **IDENT** can never collide with a keyword; the scanner checks the closed tables before emitting **IDENT**.

### 4.2.4 Reserved words and mode literals

Reserved words (case-sensitive):
```
athlete  workout  exercise  meal  progress  target  goal  plan  week
with     workouts meals     let   bodyweight
sets     reps     weight    protein  carbs   fat
per      of       and       when  else  routine  use
```

Mode literals: `bulk cut maintain` (goal modes); `lose gain` (weight-change directions). Flagged with their own token class so that misuse in non-positional contexts is a parse error, not a semantic one.

### 4.2.5 Units

| Family | Lexemes |
|---|---|
| Mass | `g`, `kg`, `lb` |
| Energy | `kcal`, `kJ` |
| Time | `s`, `min`, `h`, `day`, `week` |

`week` is dual: a reserved word in `plan week` and a unit in `2.5 kg/week`; the scanner emits one token and the parser disambiguates by position. Compound rates (`kg/week`) are three tokens (`NUM_LIT UNIT "/" UNIT`); the parser fuses them in `<rate>`.

## 4.3 Syntax (EBNF) **[P1]**

### 4.3.1 Notation

Sebesta §3.3 EBNF: `→` defines, `|` separates alternatives, `{ X }` = zero-or-more, `[ X ]` = optional. Terminals in `"double quotes"` or uppercase token-class names (§4.2.2). Start symbol: `<program>`.

### 4.3.2 Productions

```ebnf
(* — top level — *)
<program>          → <athlete-block>
<athlete-block>    → "athlete" "(" <athlete-params> ")"
                     "{" { <routine-decl> } <body-sections>
                         <rules-section> <schedule-stmt> "}"
<athlete-params>   → "bodyweight" ":" <quantity> "," "goal" ":" <goal-mode>
<goal-mode>        → "bulk" | "cut" | "maintain"

(* — body: at least one of plan / intake — *)
<body-sections>    → <plan-section> [ <intake-section> ] | <intake-section>
<plan-section>     → <workout-decl> { <workout-decl> }
<workout-decl>     → "workout" STRING_LIT
                     "{" <workout-item> { <workout-item> } [ <progress-stmt> ] "}"
<workout-item>     → <exercise-decl> | <routine-call>
<exercise-decl>    → "exercise" STRING_LIT
                     "{" "sets" ":" NUM_LIT ","
                         "reps" ":" NUM_LIT ","
                         "weight" ":" <quantity> "}"
<progress-stmt>    → "progress" "weight" <rate>
<routine-decl>     → "routine" STRING_LIT "(" <routine-params> ")"
                     "{" <exercise-decl> { <exercise-decl> } [ <progress-stmt> ] "}"
<routine-params>   → IDENT ":" UNIT { "," IDENT ":" UNIT }
<routine-call>     → "use" STRING_LIT "(" <quantity> { "," <quantity> } ")"

(* — intake — *)
<intake-section>   → <meal-decl> { <meal-decl> }
<meal-decl>        → "meal" STRING_LIT "{" <macro-decl> { <macro-decl> } "}"
<macro-decl>       → <macro-name> ":" <quantity>
<macro-name>       → "protein" | "carbs" | "fat"

(* — rules — *)
<rules-section>    → { <rule-decl> }
<rule-decl>        → <target-decl> | <goal-rate-decl> | <let-decl> | <when-decl>
<target-decl>      → "target" <macro-name> <rel-op> <quantity>
                         [ "per" UNIT "of" "bodyweight" ]
<goal-rate-decl>   → "goal" <direction> <rate>
<direction>        → "lose" | "gain"
<let-decl>         → "let" IDENT "=" <expression>

(* — selection statement: matching branch applies; dead branch discarded — *)
<when-decl>        → "when" "goal" ":" <goal-mode>
                         "{" { <rule-decl> } "}"
                         [ "else" "{" { <rule-decl> } "}" ]

(* — schedule — *)
<schedule-stmt>    → "plan" [ NUM_LIT ] "week" "with" <schedule-clauses>
<schedule-clauses> → "workouts" ":" <name-list> [ "," "meals" ":" <name-list> ]
                   | "meals"    ":" <name-list>
<name-list>        → "[" STRING_LIT { "," STRING_LIT } "]"

(* — expressions and quantities — *)
<quantity>         → NUM_LIT UNIT
<rate>             → NUM_LIT UNIT "/" UNIT
<expression>       → <add-expr>
<add-expr>         → <mul-expr>  { ("+" | "-") <mul-expr>  }
<mul-expr>         → <unary>     { ("*" | "/") <unary>     }
<unary>            → [ "-" ] <atom>
<atom>             → NUM_LIT [ UNIT [ "/" UNIT ] ]   (* scalar, quantity, or rate *)
                   | IDENT
                   | "(" <expression> ")"
<rel-op>           → ">=" | "<=" | ">" | "<" | "==" | "!="
```

### 4.3.3 Grammar properties

**Program shape.** `<program> → <athlete-block>` admits exactly one athlete context. `<body-sections>` uses one-or-more productions (never `{ }`) so an empty body is a parse error with a domain-specific message rather than a semantic error. `<schedule-stmt>` is mandatory and last.

**LL(1) class (Sebesta §4.4).** Every alternative is selected by a unique reserved word in its FIRST set: `FIRST(<exercise-decl>) = { exercise }`, `FIRST(<routine-call>) = { use }`, `FIRST(<when-decl>) = { when }`, `FIRST(<plan-section>) = { workout }`, `FIRST(<intake-section>) = { meal }`, etc. No left recursion; repetition uses EBNF `{ }`. One-token look-ahead suffices everywhere — matching the recursive-descent implementation in D2 (Sebesta §4.4.1).

**No forward references.** `{ <routine-decl> }` precedes `<body-sections>` syntactically; `<schedule-stmt>` follows all declarations. A `<let-decl>` may only reference `IDENT`s introduced by earlier `<let-decl>`s — a static check enforced by the type checker.

## 4.4 Semantics **[P2]**

### 4.4.1 Style

Semantics is given in **big-step operational** style (Sebesta §3.5.1 — Plotkin/Kahn natural semantics). Evaluation judgements have the form `env ⊢ node ⇓ value`; declaration judgements `env ⊢ decl ⇝ env'`. FitLang is pure and one-shot — no input, no I/O, no mutation — so evaluation order is fixed (top-down, left-to-right) but observationally irrelevant.

The denotation of any well-typed program is a `Plan`:

```
Plan = { athlete : { bodyweight : Mass, goal : GoalMode,
                     goalRate   : Maybe Rate(Mass, Time) },
         weeks   : [ Week ] }                          -- length N ≥ 1

Week = { index : Number,  projectedBodyweight : Mass,
         workouts : [ EvaluatedWorkout ],  meals : [ EvaluatedMeal ],
         targets  : [ TargetReport ] }
```

### 4.4.2 Construct 1 — `let` bindings and expressions

**Atomic forms:**

```
env ⊢ NUM_LIT n              ⇓  n : Number
env ⊢ NUM_LIT n UNIT u       ⇓  canonicalize(n, u) : Quantity(family u)
env ⊢ NUM_LIT n UNIT u1/u2   ⇓  canonicalize(n,u1) / canonicalize(1,u2) : Rate(family u1, family u2)
env ⊢ IDENT x                ⇓  env(x)
```

`canonicalize(n, u)` rescales `n` to the canonical unit of its family (`kg → g`, `week → s`, `kcal → kcal`). All arithmetic in the interpreter operates on canonicalized values.

**Expression rule:**

```
env ⊢ e₁ ⇓ v₁    env ⊢ e₂ ⇓ v₂    v₁ ⊕ v₂ defined per §4.5
——————————————————————————————————————————————————————————
env ⊢ e₁ "⊕" e₂  ⇓  v₁ ⊕ v₂
```

The `defined per §4.5` premise is discharged statically by the type checker; the interpreter never encounters an undefined arithmetic case at run time.

**`let` declaration (LET):**

```
        env ⊢ e ⇓ v
————————————————————————————————
env ⊢ let IDENT x = e  ⇝  env ∪ { x ↦ v }
```

By §4.6.3's no-shadowing rule, `x ∉ dom(env)` is required — enforced statically, so the rule is unconditional at run time.

### 4.4.3 Construct 2 — `plan N week` schedule

`<schedule-stmt>` is the sole output-producing construct. It selects a horizon N (default 1 when the optional `NUM_LIT` is absent) and assembles one `Week` per index, re-evaluating workouts and targets under their declared rates:

```
bw₀  = env.bodyweight       G = goalRate from env (or ⊥)

for each k ∈ 1..N:
    bw_k  = bw₀ + G × ((k − 1) week)            -- bw₀ when G = ⊥
    W_k   = [ evalWorkout(L, k)    for L in workouts_list ]
    T_k   = [ evalTarget(t, bw_k)  for t in targets_in_scope ]
    Week_k = { index=k, projectedBodyweight=bw_k, workouts=W_k, ... }
——————————————————————————————————————————————————————————————  (PLAN)
env ⊢ plan [N] week with workouts:[...], meals:[...]
        ⇓  Plan { athlete={bw₀,goal,G}, weeks=[Week_1,...,Week_N] }
```

Inside `evalWorkout`, each exercise's weight at week k is:
`weight_k = w₀ + progressRate × ((k − 1) week)`, where `w₀` is the base weight and `Rate(Mass,Time) × Quantity(Time) → Quantity(Mass)` is §4.5.4 rule 4.

Routine calls (`use "R"(a₁,...,aₙ)`) are expanded by substituting argument values for parameter names in the routine body — static scoping: the body sees the athlete-scope environment at the routine's *declaration* site, not the call site (Sebesta §5.5.1).

Labels in the schedule that name no declaration are a **static error** (caught before run time). The interpreter also aborts at run time if a computed weight becomes negative (domain invariant: a training plan never asks for a negative bar load) or if projected bodyweight falls to zero or below over the horizon.

### 4.4.4 Construct 3 — `when` selection statement

`<when-decl>` is a **selection statement** (Sebesta §8.2) that dispatches on the athlete's goal mode. Because the dispatch value is the literal bound in `<athlete-params>`, the branch is decided at compile time (a static selector — Sebesta §8.2.1.2):

```
goal-mode in env = M
——————————————————————————————  (WHEN-then)
env ⊢ when goal: M { R₁ } else { R₂ }   ↦   R₁

goal-mode in env ≠ M
——————————————————————————————  (WHEN-else)
env ⊢ when goal: M { R₁ } else { R₂ }   ↦   R₂   (or ∅ if no else)
```

The surviving rule list is spliced into the enclosing `<rules-section>` and evaluated by the §4.4.2 rules. No run-time dispatch cost; the dead branch is eliminated before the semantic phase runs.

## 4.5 Type System **[P2]**

### 4.5.1 Primitive types and value sets

FitLang has a closed type universe — the user cannot introduce new types:

| Category | Types | Value set |
|---|---|---|
| **Scalar** | `Number` | All finite IEEE 754 doubles (§6.2.1.2) |
| **Quantity** | `Mass` (canonical `g`), `Energy` (canonical `kcal`), `Time` (canonical `s`) | Non-negative real multiples of the canonical unit |
| **Rate** | `Rate(D₁, D₂)` for families D₁, D₂ | Real multiples of canonical D₁/D₂ |
| **Mode** | `GoalMode = {bulk, cut, maintain}`, `Direction = {lose, gain}` | Finite enumeration (Sebesta §6.4) |
| **Record kinds** | `Exercise`, `Meal`, `Workout`, `Routine` | Schema-fixed field tuples (§6.7) |

Every `IDENT` bound by `<let-decl>` has type `Number`, `Quantity D`, or `Rate(D₁,D₂)`. Record kinds are not let-bindable — they are already named by their `STRING_LIT` label (Sebesta §5.8 named-constants principle: name what is computed arithmetically).

### 4.5.2 Strong typing (Sebesta §6.14)

FitLang is **strongly typed**: every type error is caught at compile time and no operation silently coerces across dimensional families. This matches Sebesta §6.14's definition — type errors are *always* detected, here statically. The same trade-off Sebesta credits Java for in §1.6: mandatory compile-time checks eliminate the class of plan-corrupting bugs (`protein: 80 kcal`, `progress weight 2.5 kcal/week`) before the lifter sees a nonsensical training week.

### 4.5.3 Type equivalence — name equivalence (Sebesta §6.15)

Two values have equivalent types iff their type expressions are spelled identically in §4.5.1. Two `Quantity` values are the same type iff they share the same dimensional family. Two record values are the same type iff they share the same kind keyword. This is **name equivalence** (Sebesta §6.15) — Ada and Pascal's approach. Cost: a hypothetical future kind with the same field schema as `Exercise` would not interoperate with it. Payoff: "same kind ⇒ same type" is a one-sentence rule requiring no field-by-field comparison. Because the grammar pins each kind to one schema, name and structural equivalence agree on every current program; we pick name because it is simpler to specify.

### 4.5.4 Implicit type coercion (Sebesta §7.4)

One implicit conversion is permitted: **within-family unit rescaling to the canonical unit** (`78 kg → 78000 g`, `1 lb → 453.6 g`). This is a widening coercion (Sebesta §7.4.1) — always lossless within the value set, never capable of hiding a bug. All cross-family operations are static errors: `80 kg + 200 kcal` does not type-check. Arithmetic rules:

1. `Quantity D ± Quantity D → Quantity D` (same family — within-family coercion applied first)
2. `Number × Quantity D → Quantity D` and `Quantity D / Number → Quantity D` (scalar scaling)
3. `Rate(D, Time) × Quantity Time → Quantity D` (dimensional cancellation — used by `progress` and `goal`)
4. All other combinations are static type errors (including `Quantity D × Quantity D`, which would yield `D²`, absent from the type universe)

### 4.5.5 Structured type design (Sebesta §6.7)

The four record kinds (`Exercise`, `Meal`, `Workout`, `Routine`) follow the **fixed-schema record** design of Sebesta §6.7: heterogeneous fields identified by name, accessed via keyword notation (like Ada aggregate syntax), laid out in declaration order. Key design decisions:

- **Schema is grammar-enforced**, not user-declared. Missing or extra fields are syntax errors; the type checker only verifies field-level types (e.g., `weight:` slot must be `Quantity(Mass)`) and routine-call arity.
- **No record variables** — record kinds are not let-bindable. Records are constructed at their declaration site and referenced by label thereafter.
- **Name equivalence on record kinds** (§4.5.3): `Exercise` and `Meal` can never have the same type even if their field layouts happened to match (they cannot, by grammar).

## 4.6 Names, Binding, Scope, Lifetime **[P1]**

### 4.6.1 Legal identifiers and name spaces (Sebesta §5.2)

FitLang has **two disjoint name spaces**:

| Name space | Lexical class | Introduced by | Used in |
|---|---|---|---|
| **Labels** | `STRING_LIT` | `workout`, `exercise`, `meal`, `routine` declarations | `<schedule-stmt>` name lists; `use "..."` |
| **Identifiers** | `IDENT` | `<let-decl>`; `<routine-params>` | `<expression>` RHS |

`IDENT` form is `[A-Za-z_][A-Za-z0-9_]*` minus the closed lexicon. Case-sensitive; ASCII only. The two spaces never interact: a `<schedule-stmt>` resolves against labels only, an `<expression>` against identifiers only — no parser ambiguity (Sebesta §5.2 separate-name-space technique).

### 4.6.2 Binding times (Sebesta §5.4)

All bindings are **static** — made at or before run time, never changed:

| Attribute | Bound at | Mechanism |
|---|---|---|
| Name → declaration site | Parse time | Unique declaration per §4.6.3 |
| Name → type | Type-check time | Inferred from RHS; no surface annotation |
| Name → value | Constant evaluation | Pure, no I/O, no side effects |

No dynamic-type binding (§5.4.2.2), no late-bound variables.

### 4.6.3 Scope — static lexical (Sebesta §5.5.1)

Scope is **static (lexical)**, block-structured with `{ }`. We reject dynamic scoping (§5.5.6) outright: it makes static analysis effectively impossible (Sebesta §5.5.7 — type checks on nonlocal references cannot be done statically) and FitLang has no first-class procedures to justify the cost.

Three rules:

1. **`let`-bound IDENTs live at athlete scope only.** Visible in every `<expression>` following the binding in the same `<athlete-block>`. No forward references; enforced statically by a single-pass type checker.
2. **Labels live at athlete scope, partitioned by kind.** No two workouts may share a label; no two meals; no two routines — but a workout and a meal *may* share one because they resolve in separate namespaces.
3. **No shadowing.** Rebinding an IDENT already in scope is a static error — stricter than C/Java (Sebesta §5.5.2), affordable because FitLang's scope nesting is a fixed three levels deep.

### 4.6.4 Lifetime (Sebesta §5.4.3)

By immutability (§4.6.5) and the absence of closures or heap allocation, **lifetime equals scope**. Every binding behaves as if statically allocated for the program's lifetime (Sebesta §5.4.3.1). Consequences: no dangling-reference class of bug (§5.4.3.3), no initialise-before-use rule, no garbage-collection story. The interpreter may inline every binding, constant-fold it, or stack-allocate it — the language definition does not distinguish.

### 4.6.5 Immutability and named constants (Sebesta §5.8)

There is no assignment operator. Every name is a **named constant** (Sebesta §5.8) — bound once at its declaration. This is Erlang-style single-assignment and Rust's `let` default. The iterative content the domain needs (week-over-week weight progression) is expressed declaratively via `progress` + `plan N week`, not by mutation.

## 4.7 Expressions & Assignment **[P2]**

### 4.7.1 Operator precedence and associativity (Sebesta §7.2)

| Level (tight → loose) | Operators | Associativity | Sebesta analogue |
|---|---|---|---|
| 1 | unary `-` | right | C, Java, Pascal |
| 2 | `*`, `/` | left | C, Java |
| 3 | binary `+`, `-` | left | C, Java |
| 4 | `>`, `<`, `>=`, `<=`, `==`, `!=` | non-associative | Pascal |

Precedence is encoded by stratifying the EBNF into `<add-expr>` → `<mul-expr>` → `<unary>` → `<atom>` (§4.3.2). Left-associativity follows from left-to-right `{ }` iteration in `<add-expr>` and `<mul-expr>` — Sebesta §3.3.2: with EBNF `{ }`, the *parser* enforces associativity.

Relational operators appear only inside `<target-decl>` and are non-associative by grammar: exactly one `<rel-op>` per target line; `a < b < c` is not writable.

### 4.7.2 Boolean operators and short-circuit evaluation (Sebesta §7.6)

FitLang has **no Boolean operators**. The only conditional construct (`<when-decl>`) dispatches on a single mode literal — there is no need for compound predicates, and therefore **no short-circuit semantics question arises**. Adding `&&`/`||` later would be a strictly additive change at precedence level 5.

### 4.7.3 Operand evaluation order (Sebesta §7.2.2)

Operand evaluation is **left-to-right** by convention. The order is observationally irrelevant: by immutability (§4.6.5) and purity (§4.4.1), no expression has functional side effects (Sebesta §7.2.2.1), so two evaluation orders that respect the no-forward-reference rule of §4.6.3 produce identical values. Implementers may reorder freely.

### 4.7.4 Assignment (Sebesta §7.7)

FitLang has **no assignment statement**. The two tokens that superficially resemble assignment are:

- `=` in `<let-decl>` — a **binding** operator (ML's `let x = e in ...`, not Pascal's `x := e`). It introduces a fresh name; illegal if the name is already bound. There is no L-value; `x` is a name, not a storage cell.
- `:` in record literals — **field initialisation**, pairing a schema-fixed field name with its value at construction time (Ada aggregate notation, Sebesta §6.7).

## 4.8 Design Rationale **[P2]**

**1. Strong typing with dimensional families (§4.5.2, §4.5.4).** We commit to compile-time rejection of every cross-family operation and disallow same-family multiplication between two quantities (which would yield `D²`, absent from the type universe). Cost: ergonomic ceremony — `80 kg + 200` is rejected; the user writes `+ 200 g`. Payoff: the entire class of unit-confusion bugs (`progress weight 2.5 kcal/week`) is eliminated before the lifter sees a single training week. This is Sebesta §1.6's reliability-vs-writability trade-off taken deliberately, as Java took it against C.

**2. Name equivalence on record kinds (§4.5.3).** We pick name over structural equivalence (Sebesta §6.15). Cost: a future kind with `Exercise`'s field layout would not interoperate with it. Payoff: the checker needs no field-by-field comparison routine; "same kind ⇒ same type" is one sentence. Because the grammar pins every kind to one schema, name and structural equivalence currently agree on every legal program — we pick name because it is the rule simpler to specify, implement, and defend.

**3. Closed type universe, no user-defined types (§4.5.1).** Three quantity families and four record kinds exhaust the type universe. Cost: extensibility — adding a kind requires a grammar change. Payoff: a linear-time single-pass checker and a closed equivalence rule, both of which depend on the type universe being finite and listable. Extensibility is not a real requirement for a course DSL.

**4. Within-family coercion only (§4.5.4).** One implicit conversion is admitted — unit rescaling to the canonical unit (`kg → g`). All else is a hard error. Cost: the user must occasionally spell out a unit they "obviously" intended, as Ada requires (Sebesta §6.14 derived-type restriction). Payoff: within-family widening coercion is always lossless (Sebesta §7.4.1), so the one permitted implicit conversion never hides a type mismatch.

**5. C/Java operator precedence and left-associativity (§4.7.1).** Unary → multiplicative → additive → relational matches the dominant convention Sebesta surveys in §7.2.1.1. Cost: a user from an APL background (all operators equal-precedence, right-to-left) would be surprised, but the domain user is a lifter, not an APL programmer. Payoff: no learning overhead for anyone who has seen C, Java, or Python; the precedence table in the documentation is sufficient. Left-associativity is a corollary: `a − b − c` means `(a − b) − c`, which is the expected reading for subtraction and consistent with every C-family language.

**6. No assignment statement, single-assignment discipline (§4.7.4, §4.6.5).** Immutability is enforced at the language level. Cost: imperative iteration (`weight = weight + 2.5 kg`) cannot be expressed directly — the user cannot mutate a variable across weeks. Payoff: lifetime equals scope (§4.6.4), no dangling references, no initialise-before-use specification, a single-pass type checker, and observationally irrelevant evaluation order (§4.7.3). The iterative content the domain needs is recovered declaratively by `progress` + `plan N week`, faithful to how a periodised training block is actually written: a base value, a rate, a horizon.
