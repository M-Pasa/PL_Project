## 4.1 Language Overview **[P1]**

**Name.** FitLang.

**Domain.** A domain-specific language for writing personal strength-training programs together with the nutrition required to execute them. A FitLang program describes (a) workout templates with sets, reps, and loads, (b) meals with macronutrients, and (c) progression and nutritional rules — all evaluated against a single lexically scoped athlete context (bodyweight, goal). Workouts and nutrition share one type system — physical quantities (`Mass`, `Energy`, macro grams, time) are first-class typed values, and the language refuses to silently mix incompatible units. The output of a FitLang program is a planned training week with per-day macro and calorie targets derived from the planned workouts and the lifter's stated goal.

**Sample program.**

```fitlang
athlete(bodyweight: 78 kg, goal: bulk) {

    workout "Push Day" {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: 80 kg }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: 50 kg }
        progress weight 2.5 kg/week
    }

    meal "Breakfast" {
        protein: 40 g
        carbs:   80 g
        fat:     15 g
    }

    target protein >= 1.8 g per kg of bodyweight
    goal   lose 0.5 kg/week

    plan week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```

**Justification (Sebesta §1.3 evaluation criteria).**
FitLang is designed to optimise **writability** for its application domain (Sebesta §1.3.2 — writability is always discussed *relative to a domain*) and **reliability** through a strict, unit-aware type system (§1.3.3). The domain user — a lifter planning a training block, not a software engineer — should be able to express a workout in roughly the same words they would write in a notebook (`sets: 4, reps: 6, weight: 80 kg`); this is the form-and-meaning correspondence Sebesta lists under readability (§1.3.1) and the closeness-to-application-domain factor that drives the *cost of writing programs* (§1.3.4). Reliability is bought by typed physical quantities that disallow arithmetic across dimension families (`80 kg + 1 m` is a static type error) and disallow same-unit multiplication or division (`kg * kg`, `kg / kg`); this is the same trade-off Sebesta credits Java for taking against C in §1.6, where mandatory checks cost execution time but eliminate a class of bugs entirely. The criteria FitLang knowingly sacrifices are **generality** (it cannot, and should not, be used outside fitness/nutrition planning — Sebesta lists generality only as a secondary criterion, §1.3) and **cost of execution** (a tree-walking Java interpreter is at least an order of magnitude slower than compiled code per §1.7's discussion of pure interpretation, but a one-second plan calculation is irrelevant for the domain).

## 4.2 Lexical Structure **[P1]**

### 4.2.1 Character set, line endings, whitespace

FitLang source is UTF-8. The lexer treats the source as a stream of Unicode code points, but the only places non-ASCII code points may appear are inside string literals and inside line comments — every other lexeme is ASCII. Whitespace (`U+0020` space, `U+0009` tab, `U+000A` LF, `U+000D` CR) is insignificant: it separates tokens but carries no other meaning. Indentation is not part of the grammar; we explicitly reject Python-style significant indentation (Sebesta §3.1) because the block syntax already uses `{ }` and double-duty would only cost readability.

### 4.2.2 Comments

A `#` outside a string literal starts a line comment that runs to the next newline. There is no block-comment form. One comment style keeps the lexer's comment automaton trivial and matches the configuration-language feel of the surface syntax.

```fitlang
# Cutting block, week 3
workout "Push Day" { ... }
```

### 4.2.3 Token classes

Every lexeme produced by the scanner falls into exactly one of the classes below. Class names in **bold** are the terminal symbols used by §4.3's EBNF.

| Class | Regex (POSIX-extended) | Examples |
|---|---|---|
| **RESERVED** | one of the listed reserved words (table 4.2.5) | `athlete`, `workout`, `let` |
| **MODE_LIT** | one of the listed mode literals (table 4.2.5) | `bulk`, `cut`, `lose` |
| **UNIT** | one of the listed unit lexemes (table 4.2.6) | `kg`, `kcal`, `week` |
| **NUM_LIT** | `[0-9]+ (\.[0-9]+)?` | `80`, `2.5`, `0.5` |
| **STRING_LIT** | `"` *( any Unicode except `"` and newline; `\\` and `\"` escapes )* `"` | `"Push Day"`, `"Kahvaltı"` |
| **IDENT** | `[A-Za-z_][A-Za-z0-9_]*` minus any RESERVED / MODE_LIT / UNIT lexeme | `bodyweightTarget`, `dailyKcal` |
| **PUNCT** | one of `{ } [ ] ( ) , : / + - * = > < >= <= == !=` | `{`, `>=`, `:` |

Numeric literals carry no sign — leading `-` is the unary minus operator, lexed as the `-` punctuation token. Exponent form (`1.5e3`) is rejected; the domain has no quantity that benefits from it. There is no integer / float distinction at the lexical level — both `80` and `80.0` produce a single **NUM_LIT** token whose attribute is the numeric value; the type system (§4.5) decides where each is admissible.

### 4.2.4 String literals

A string literal is delimited by ASCII double quotes. Inside the literal any Unicode code point is permitted except an unescaped `"` or raw newline (literal newlines in source are rejected to keep error recovery line-local). Three escape sequences are recognized: `\"` (literal quote), `\\` (literal backslash), and `\n` (line feed). Octal, hex, and `\u` escapes are not supported — labels only need a way to embed quotes, backslashes, and the occasional line break.

### 4.2.5 Reserved words and mode literals

FitLang uses a **closed lexicon**: every unquoted alphabetic lexeme is either a reserved word, a mode literal, or a unit. The user introduces names only via string literals (workout / exercise / meal labels) or via numeric `let` bindings (one **IDENT** class, restricted by §4.5 to scalar and quantity types). This is the "fully reserved" stance Sebesta credits to Pascal in §3.1 — a reserved word can never be redeclared as an identifier, eliminating the keyword-vs-identifier ambiguity ALGOL 60 left to context.

Reserved words (case-sensitive):

```
athlete  workout  exercise  meal  progress  target  goal  plan  week
with     workouts meals     let   bodyweight
sets     reps     weight    protein  carbs   fat
per      of       and
when     else     routine   use
```

The last row introduces the §2 control structure (`when` / `else`) and the §2 user-defined function template (`routine` declaration, `use` call). They participate in the closed lexicon on the same terms as every other reserved word — no shadowing, no use as `IDENT`.

Mode literals (case-sensitive — finite enumerations consumed positionally in §4.3):

```
bulk  cut  maintain         # goal modes
lose  gain                   # weight-change directions
```

Mode literals are syntactically distinct from reserved words because §4.3 admits them only in fixed positions (`goal: <mode>`, `<direction> <quantity>`); flagging them with their own token class catches misuse at parse time rather than via a semantic check.

### 4.2.6 Units

Units are recognized as their own token class so that the parser can require a **NUM_LIT** **UNIT** pair anywhere a quantity is expected (per the §1 design call: lex two tokens, pair at parse time). Each unit lexeme carries an attribute identifying its dimension family — Mass, Energy, Time — used by §4.5's coercion rules.

| Family | Lexemes |
|---|---|
| Mass | `g`, `kg`, `lb` |
| Energy | `kcal`, `kJ` |
| Time | `s`, `min`, `h`, `day`, `week` |

Macronutrients (protein, carbs, fat) are themselves masses — `protein: 80 g` and `bodyweight: 78 kg` carry quantities of the same family, and within-family coercion (§4.5) means `bodyweight: 70000 g` is legal and equal to `bodyweight: 70 kg`. Three families instead of four keeps the type rule simple and lets the cross-family error rule (`80 kg + 200 kcal` is a static type error) carry its weight without artificial splits. `week` is both a reserved word (in `plan week`) and a unit (in `2.5 kg/week`); the scanner emits the same lexeme and §4.3 distinguishes by position. Compound rate forms like `kg/week` are *not* a single lexeme — the scanner emits three tokens (`NUM_LIT` `UNIT` `/` `UNIT`) and the parser builds the rate.

### 4.2.7 Punctuation

The complete punctuation alphabet is `{`, `}`, `[`, `]`, `(`, `)`, `,`, `:`, `/`, `=`, `>`, `<`, `>=`, `<=`, `==`, `!=`, `+`, `-`, `*`. Maximal-munch (Sebesta §4.2 longest-match rule) disambiguates `>=`/`<=`/`==`/`!=` against their single-character prefixes; the scanner reads as many characters as form a valid punctuation lexeme before emitting a token. Equality (`==`) and inequality (`!=`) are reserved here at the lexical level; their semantics — operand types, whether they compare quantities by canonicalized value or by surface unit — are specified in §4.7 (Part 2). There is no `===`: FitLang has no reference-typed values, so the JavaScript-style value-vs-identity distinction would be ceremony with no payoff. There is no `;` — newlines and `,` carry list and statement separation.

### 4.2.8 Identifiers

**IDENT** is `[A-Za-z_][A-Za-z0-9_]*` minus the closed lexicon of §4.2.5–4.2.6. ASCII only — Unicode identifiers add a normalization burden (NFC vs NFD homoglyphs) Sebesta calls out in §3.1, and the DSL has no payoff. Case-sensitive: `bodyweight` (the reserved word) and `Bodyweight` (a legal **IDENT**) are distinct lexemes; we follow the C/Java convention and reject Fortran-style case folding (Sebesta §3.1 lists case-insensitivity as a historical outlier). The scanner enforces the "minus reserved" rule by checking each candidate identifier against the reserved/mode/unit tables before emitting **IDENT**, so the parser never has to disambiguate.

## 4.3 Syntax (EBNF) **[P1]**

### 4.3.1 Notation

The grammar is written in the extended BNF dialect Sebesta uses in §3.3: `→` defines a nonterminal, `|` separates alternatives, `{ X }` denotes zero-or-more repetition, `[ X ]` denotes an optional element. Nonterminals are written in `<angle-brackets>`. Terminals are either literal lexemes in `"double quotes"` (matching the surface syntax verbatim) or the uppercase token-class names introduced in §4.2.3 (`NUM_LIT`, `STRING_LIT`, `IDENT`, `UNIT`). The start symbol is `<program>`.

### 4.3.2 Productions

```ebnf
(* — top level — *)
<program>          → <athlete-block>

<athlete-block>    → "athlete" "(" <athlete-params> ")"
                     "{"
                         { <routine-decl> }
                         <body-sections>
                         <rules-section>
                         <schedule-stmt>
                     "}"

<athlete-params>   → "bodyweight" ":" <quantity> "," "goal" ":" <goal-mode>
<goal-mode>        → "bulk" | "cut" | "maintain"

(* — at least one of plan / intake; both allowed in fixed order — *)
<body-sections>    → <plan-section> [ <intake-section> ]
                   | <intake-section>

(* — training plan section: one or more workouts — *)
<plan-section>     → <workout-decl> { <workout-decl> }

<workout-decl>     → "workout" STRING_LIT
                     "{"
                         <workout-item> { <workout-item> }
                         [ <progress-stmt> ]
                     "}"

(* — a workout body is built from inline exercise records or
     calls to a previously-declared routine template — *)
<workout-item>     → <exercise-decl> | <routine-call>

(* — <exercise-decl> is the §2 structured type (record schema):
     a fixed set of named, typed fields evaluated positionally — *)
<exercise-decl>    → "exercise" STRING_LIT
                     "{"
                         "sets"   ":" NUM_LIT ","
                         "reps"   ":" NUM_LIT ","
                         "weight" ":" <quantity>
                     "}"

<progress-stmt>    → "progress" "weight" <rate>

(* — user-defined parameterised template: a named routine,
     declared once at athlete scope, instantiated at any
     workout body via <routine-call>.  Satisfies §2's
     "user-defined functions with parameters" requirement
     in a declarative DSL idiom — the routine is not a
     procedure but a parameterised exercise group. — *)
<routine-decl>     → "routine" STRING_LIT "(" <routine-params> ")"
                     "{"
                         <exercise-decl> { <exercise-decl> }
                         [ <progress-stmt> ]
                     "}"
<routine-params>   → IDENT ":" UNIT { "," IDENT ":" UNIT }
<routine-call>     → "use" STRING_LIT "(" <quantity> { "," <quantity> } ")"

(* — intake section: one or more meals — *)
<intake-section>   → <meal-decl> { <meal-decl> }

<meal-decl>        → "meal" STRING_LIT
                     "{"
                         <macro-decl> { <macro-decl> }
                     "}"

<macro-decl>       → <macro-name> ":" <quantity>
<macro-name>       → "protein" | "carbs" | "fat"

(* — rules section: zero or more nutritional/progression rules — *)
<rules-section>    → { <rule-decl> }
<rule-decl>        → <target-decl> | <goal-rate-decl> | <let-decl> | <when-decl>

<target-decl>      → "target" <macro-name> <rel-op> <quantity>
                         [ "per" UNIT "of" "bodyweight" ]
<goal-rate-decl>   → "goal" <direction> <rate>
<direction>        → "lose" | "gain"
<let-decl>         → "let" IDENT "=" <expression>

(* — control structure (§2 requirement): rules inside the
     "then" block apply only when the athlete's declared
     goal mode matches; the optional "else" branch carries
     the rules that apply otherwise.  The dispatch value
     is the goal mode bound in <athlete-params>, so the
     conditional is fully decidable at compile time
     (a static <if> in Sebesta §8.1 terms) — there is no
     run-time predicate. — *)
<when-decl>        → "when" "goal" ":" <goal-mode>
                         "{" { <rule-decl> } "}"
                         [ "else" "{" { <rule-decl> } "}" ]

(* — schedule: exactly one, closes the program;
     at least one of workouts / meals must appear — *)
<schedule-stmt>    → "plan" "week" "with" <schedule-clauses>
<schedule-clauses> → "workouts" ":" <name-list> [ "," "meals" ":" <name-list> ]
                   | "meals"    ":" <name-list>
<name-list>        → "[" STRING_LIT { "," STRING_LIT } "]"

(* — quantities, rates, expressions — *)
<quantity>         → NUM_LIT UNIT
<rate>             → NUM_LIT UNIT "/" UNIT

(* P1 expression layer: additive operators only.  Multiplicative
   operators ("*", "/" outside of <rate>), parenthesised grouping,
   and the full precedence/associativity table are added in §4.7
   (Part 2).  "+" and "-" are left-associative; with no other
   operators present in P1 there is no precedence question to
   answer here.  Operand-type rules (only same-family quantities
   may be added; cross-family is a static type error) live in §4.5. *)
<expression>       → <term> { <add-op> <term> }
<term>             → <quantity> | NUM_LIT | IDENT
<add-op>           → "+" | "-"

<rel-op>           → ">=" | "<=" | ">" | "<" | "==" | "!="
```

### 4.3.3 What the start symbol enforces

The grammar is *structured* in the Pascal sense (Sebesta §3.3, Pascal program template): a FitLang program is not a flat list of declarations but a fixed skeleton of sections in a required order — **header → plan → intake → rules → schedule**. Three consequences follow at parse time, before any semantic analysis runs:

1. **Every program has exactly one athlete context.** `<program> → <athlete-block>` admits no alternative; a file containing two `athlete(...)` blocks or none fails the LL(1) prediction set on the first token. This is the rule the §4.5 type system relies on when it resolves `bodyweight` without a scope-lookup step.
2. **An empty body, or a body without a schedule, is unparseable.** `<body-sections>` requires at least one of plan / intake — written as a two-alternative production rather than `[ <plan-section> ] [ <intake-section> ]`, which would admit the empty derivation. Each section that *is* present must be non-empty (one-or-more, written `<workout-decl> { <workout-decl> }`, never `{ <workout-decl> }`). `<schedule-stmt>` is mandatory and is the last child of `<athlete-block>`. The parser therefore rejects three classes of malformed program with one token of look-ahead and a domain-specific message: empty athlete blocks, sections declared but empty (e.g., `meal "Breakfast" {}`), and programs that never schedule themselves. Workout-only and meal-only programs *are* legal — a diet-only client and a strength-only client are both first-class FitLang programs.
3. **Rules reference an existing goal mode.** `<athlete-params>` fixes the goal mode (`bulk` / `cut` / `maintain`) before any `<rule-decl>` is parsed, so `<goal-rate-decl>` in §4.4's semantics can be checked for consistency with the header (e.g., `goal: bulk` + `goal lose 0.5 kg/week` is a static contradiction) without any forward reference. The `<when-decl>` control structure piggy-backs on the same property: because the goal mode is fixed before any rule is parsed, the `when goal: <mode>` dispatch is decidable at compile time and the unused branch can be eliminated before §4.4's semantic phase even runs.
4. **Routines are declared before they are used.** `{ <routine-decl> }` appears at the head of `<athlete-block>`, strictly before `<body-sections>`. A `<routine-call>` inside a `<workout-decl>` therefore always refers to a routine name introduced earlier in the same block — the same no-forward-reference discipline §4.6.3 imposes on `let`. The grammar makes this a syntactic property; semantics need only check argument arity and unit families against the matching `<routine-decl>`.

This is the originality argument restated as a grammar property: a general-purpose language (C, Python) cannot bake its program shape into the start symbol because it does not know what a program is *for*; FitLang does, so it should.

### 4.3.4 Lexical / syntactic boundary

Two design calls are visible in the productions above and worth flagging because they push work across the lex/parse boundary:

- **Quantities are pairs at the parse level, not at the lex level.** `<quantity> → NUM_LIT UNIT` follows the §4.2.6 decision: the scanner emits two tokens, the parser fuses them. The alternative — a single `QUANTITY_LIT` lexeme matching `[0-9]+(\.[0-9]+)?[a-zA-Z]+` — would force the scanner to know the unit family table and would re-couple lexical and semantic concerns Ada deliberately separates.
- **Compound rates (`kg/week`) are three tokens, not one.** `<rate> → NUM_LIT UNIT "/" UNIT` reuses the existing `/` punctuation rather than introducing a `RATE_LIT` lexeme. The parser distinguishes division-of-quantities (a §4.7 expression form, P2) from rate construction by position: a `/` immediately following a `NUM_LIT UNIT` pair inside a `<progress-stmt>` or `<goal-rate-decl>` constructs a rate, anywhere else it is the binary operator.

### 4.3.5 Parsing class

The grammar is **LL(1)** as written: every alternative inside `<rule-decl>` and `<athlete-block>`'s body is selected by a unique reserved word in the FIRST set (`workout`, `meal`, `target`, `goal`, `let`, `when`, `progress`, `plan`), and the one-or-more constructs (`<plan-section>`, `<intake-section>`) commit on their distinguishing first token (`workout`, `meal`). The two alternative productions added to support diet-only and workout-only programs — `<body-sections>` and `<schedule-clauses>` — are also LL(1) by inspection: `FIRST(<plan-section>) = { workout }` is disjoint from `FIRST(<intake-section>) = { meal }`, and the two alternatives of `<schedule-clauses>` are selected by `workouts` vs `meals`. No left recursion appears; the only repetition is right-recursive in EBNF form. This matches the recursive-descent implementation strategy planned for D2 and the parsing-class discussion Sebesta gives in §4.4 — LL(1) is the cheapest grammar class that buys us readable hand-written error messages and predictable single-pass parsing.

The §2 extensions preserve LL(1) by construction. Each new alternative is selected by a reserved word disjoint from every existing FIRST set:

- `FIRST(<routine-decl>) = { routine }`, disjoint from `FIRST(<plan-section>) = { workout }` and `FIRST(<intake-section>) = { meal }`. The optional `{ <routine-decl> }` prefix in `<athlete-block>` therefore commits or skips on the first token after `{` with a single look-ahead.
- Inside a workout body, `FIRST(<exercise-decl>) = { exercise }` and `FIRST(<routine-call>) = { use }` are disjoint, so `<workout-item>` is LL(1).
- `FIRST(<when-decl>) = { when }` is disjoint from the existing `<rule-decl>` alternatives' FIRSTs (`{ target, goal, let }`), so adding it as a fourth alternative is direct. The optional `else`-branch is selected by `{ else }`, which does not appear in the FOLLOW set of `<when-decl>`'s body (the body is closed by `}`, after which only another `<rule-decl>` start, `plan`, or the `else` look-ahead can follow — and the closing `}` consumes the choice deterministically before `else` is offered).
- `FIRST(<routine-params>) = { IDENT }` is the only IDENT-headed production in `<athlete-block>`, but it appears only inside a `(` ... `)` pair already committed by the preceding `routine STRING_LIT (`, so no ambiguity arises with `<expression>`'s `<term> → IDENT` alternative (which is reachable only through `<let-decl>`).

### 4.3.6 Sample derivation

For the §4.1 sample program, the leftmost derivation begins:

```
<program>
⇒ <athlete-block>
⇒ "athlete" "(" <athlete-params> ")" "{" <plan-section> <intake-section> <rules-section> <schedule-stmt> "}"
⇒ "athlete" "(" "bodyweight" ":" <quantity> "," "goal" ":" <goal-mode> ")" "{" ...
⇒ "athlete" "(" "bodyweight" ":" NUM_LIT UNIT "," "goal" ":" "bulk" ")" "{" ...
⇒ "athlete" "(" "bodyweight" ":" 78 kg "," "goal" ":" "bulk" ")" "{" ...
```

A full parse tree is given in D3's worked example (Part 1 deliverable, three valid programs). The point of the partial derivation here is to show that the first six tokens of any FitLang program are forced by the grammar — `athlete ( bodyweight : NUM_LIT UNIT` — which is what gives D2's parser its strong front-end error recovery story.

## 4.6 Names, Binding, Scope, Lifetime **[P1]**

### 4.6.1 What is nameable

FitLang has **two disjoint name spaces**, separated at the lexical level so the parser never has to disambiguate:

| Name space | Lexical class | Introduced by | Used in |
|---|---|---|---|
| **Labels** | `STRING_LIT` | `workout "..." { ... }`, `exercise "..." { ... }`, `meal "..." { ... }`, `routine "..." (...) { ... }` | the `<name-list>` of `<schedule-stmt>`; the head of a `<routine-call>` (`use "..."`) |
| **Identifiers** | `IDENT` | `<let-decl>` (athlete scope); `<routine-params>` (routine scope, parameter names) | right-hand side of any `<expression>`; `<quantity>` slots inside the routine body (P2) |

Workouts, exercises, and meals are addressed by their human-readable **string labels** (`"Push Day"`, `"Bench Press"`); these are the names the lifter or dietitian writes in a notebook, and the surface syntax preserves that — there is no separate "give your workout an internal identifier" step. Only `let`-bound numeric or quantity values get an `IDENT`, and the closed-lexicon rule of §4.2.5 guarantees an `IDENT` can never collide with a reserved word, mode literal, or unit. Sebesta §5.2 lists separate name spaces (Common Lisp's function-vs-value distinction) as a deliberate technique to enlarge the usable vocabulary without ambiguity; FitLang takes the same option for the same reason.

The two name spaces never interact: a `<schedule-stmt>` resolves its `<name-list>` against the label space only, and an `<expression>` resolves an `IDENT` against the identifier space only. There is consequently no `f(x)`-vs-`f` ambiguity, no `(name)` hack to disambiguate one from the other.

### 4.6.2 Binding times

Every binding in a FitLang program is fixed by **static binding time** in the sense of Sebesta §5.4.1 — bindings are made at compile time and do not change at run time:

| Attribute | Bound at | Mechanism |
|---|---|---|
| Name → declaration site | Compile time (parse) | The single `<workout-decl>` / `<meal-decl>` / `<let-decl>` that introduces the name |
| Name → type | Compile time (type check, §4.5 in P2) | Inferred from the right-hand side; declared types are not part of the surface syntax |
| Name → value | Compile time (constant evaluation) for all `<let-decl>`s whose right-hand side is closed under §4.7's expressions; otherwise at the start of run time | Pure declarative evaluation — no input, no I/O, no side effects |
| Name → storage | N/A | There is no observable storage: the interpreter is free to inline every name (Sebesta §5.4.3.4 *static* category, broadened to all bindings) |

FitLang has no analogue of Sebesta §5.4.2's *dynamic* binding times — no late-bound types, no first-class environments, no `eval`. This is a deliberate restriction: the domain user writes a plan once and runs it once, and every error the interpreter can move to compile time is one fewer surprise during the lifter's training week.

### 4.6.3 Scope

Scope is **static (lexical) and block-structured**, demarcated by the `{ ... }` brackets of `<athlete-block>`, `<workout-decl>`, `<exercise-decl>`, and `<meal-decl>`. We follow the C/Java/Pascal lineage Sebesta surveys in §5.5.1 and reject dynamic scoping (§5.5.2 — Sebesta credits it to early Lisps and notes it makes static analysis effectively impossible) outright: the DSL has no first-class procedures and no use case that would justify the cost.

The scope nesting is shallow and fixed by the grammar:

```
athlete-block
├── (let-decls, target-decls, goal-rate-decls)   ← athlete scope
└── workout-decl
    ├── exercise-decl                             ← exercise scope
    └── progress-stmt
        meal-decl                                 ← meal scope
        (macro-decls)
```

Three concrete rules follow:

1. **`let`-bound `IDENT`s live at athlete scope and only at athlete scope.** The grammar admits `<let-decl>` only as a `<rule-decl>` inside `<rules-section>`, never inside a `<workout-decl>` or `<meal-decl>`. A let-binding is therefore visible in every `<expression>` that follows it within the same `<athlete-block>` and invisible outside that block (a moot point, since there is exactly one). This is Sebesta §5.7's *referencing environment* in its simplest possible form: a flat chain of one block.
2. **Labels live at athlete scope, partitioned by kind.** Within an `<athlete-block>` no two `<workout-decl>`s may share a `STRING_LIT`, no two `<meal-decl>`s may share one, and no two `<routine-decl>`s may share one — but a workout, a meal, and a routine *may* coexist with the same label (`workout "Recovery"`, `meal "Recovery"`, `routine "Recovery"`) because each is resolved against its own kind: `<schedule-stmt>` against workout/meal labels, `<routine-call>`'s `use "..."` against the routine label space. Exercise labels live at the *workout* scope (and at the *routine* scope inside `<routine-decl>`): `"Bench Press"` may appear in two different workouts as two distinct exercise declarations. Routine parameter names (`load`, `accessory_load`, ...) live at the *routine* scope and are visible only inside that routine's body. This matches Pascal's record-field rule (Sebesta §5.5.5) — field names are scoped to their record, not globally — and the standard procedure-parameter scoping Sebesta surveys in §5.5.
3. **Forward references are forbidden.** A `<let-decl>` may only reference `IDENT`s introduced by an earlier `<let-decl>`; a `<schedule-stmt>` may only reference labels introduced by earlier `<workout-decl>`s and `<meal-decl>`s. The grammar already places `<schedule-stmt>` last, so the schedule-side of the rule is automatic; the let-side is a static check. This rules out mutual recursion among `let` bindings, which is exactly what we want — there is no recursion in the domain, and the no-forward-reference rule lets the type checker run in a single linear pass over the AST.

**No shadowing.** Re-binding an `IDENT` already in scope is a static error. With immutable bindings and a single flat scope for `let`, shadowing buys nothing and only invites the "which `dailyKcal` did I mean?" class of bug Sebesta flags in §5.5.4. (This is stricter than C/Java, which permit inner scopes to shadow outer ones; we are at liberty to be stricter because FitLang's nesting is a fixed three levels deep.)

### 4.6.4 Lifetime

Because every binding is immutable (§4.6.6 below) and there are no first-class procedures, closures, or heap allocations, **lifetime equals scope**. Sebesta §5.4.3 distinguishes four storage-binding categories — static, stack-dynamic, explicit-heap-dynamic, implicit-heap-dynamic — and FitLang collapses to a single one: every binding behaves *as if* statically allocated for the duration of the program, because no observable behaviour can distinguish that strategy from any other when nothing is mutated and the program is one-shot. The interpreter is free to allocate let-values on the stack, intern them, or constant-fold them away; the language definition does not require the user to know which.

The payoffs are concrete and worth listing for §4.8's rationale:

- No dangling-reference class of bug (Sebesta §5.4.3.3, Pascal pointer hazards) — there are no references.
- No initialise-before-use rule to specify (§5.4.3.1) — the grammar forces declaration-before-use, and the value-binding is part of the declaration.
- No garbage collection story (§5.4.3.4) — no heap.

### 4.6.5 Referencing environment

At any program point *p*, the referencing environment is the set of names visible at *p* (Sebesta §5.7). For FitLang it is:

- **Inside `<athlete-params>`:** the closed lexicon of §4.2 only — no user bindings exist yet.
- **Inside `<rules-section>` at point *p*:** the closed lexicon plus all `IDENT`s introduced by `<let-decl>`s strictly preceding *p* in the same `<rules-section>`.
- **Inside `<schedule-stmt>`:** the label space — every `STRING_LIT` introduced as a `<workout-decl>` or `<meal-decl>` head in the enclosing `<athlete-block>`.
- **Inside a `<workout-decl>` or `<meal-decl>` body:** only the closed lexicon and the constants written at the call site. (The grammar admits no `IDENT` reference inside these blocks in P1 — `weight: 80 kg` takes a literal `<quantity>`, not an `<expression>`. This is a P1-scoping decision; §4.7 in P2 will broaden the relevant fields to `<expression>` and the referencing environment will then include the athlete-scope `let`s.)

The bodyweight value declared in `<athlete-params>` is *not* an ordinary `IDENT` and does not appear in the identifier name space — it is a fixed parameter referred to by the keyword phrase `of bodyweight` in `<target-decl>`. This is the ALGOL-style approach of giving privileged values a reserved-word handle rather than an ordinary identifier (Sebesta §5.2 *special words*), and it sidesteps the question of what to do if a user wrote `let bodyweight = 90 kg`.

### 4.6.6 Immutability and named constants

There is no assignment operator (§4.7's `=` in `<let-decl>` is binding, not assignment; `:` in record literals is field initialisation). Every name in FitLang is therefore a **named constant** in Sebesta's §5.8 sense — bound once, at the declaration, and referenced thereafter without ever being updated. This is the same single-assignment discipline Erlang enforces at the language level and Rust enforces by default (`let` without `mut`).

The choice is domain-driven: a fitness program is a *plan*, not a process. Mutating "Push Day" mid-program does not correspond to anything the lifter does in the gym — if the workout changes, that is a *different* workout, and the language should make the user say so by introducing a new declaration. The cost paid for this discipline is the inability to express iterative state ("week 1 weight, week 2 weight, ..."); §4.4's `progress` construct (P2 semantics) recovers that expressivity by computing successive weeks declaratively from a base value and a rate, rather than by mutation.
