## 4.1 Language Overview **[P1]**

**Name.** FitLang.

**Domain.** A domain-specific language for writing personal strength-training programs together with the nutrition required to execute them. A FitLang program describes (a) workout templates with sets, reps, and loads, (b) meals with macronutrients, and (c) progression and nutritional rules — all evaluated against a single lexically scoped athlete context (bodyweight, goal). Workouts and nutrition share one type system — physical quantities (`Mass`, `Energy`, macro grams, time) are first-class typed values, and the language refuses to silently mix incompatible units. The output of a FitLang program is a planned training week with per-day macro and calorie targets derived from the planned workouts and the lifter's stated goal.

**Sample program.**

```fitlang
athlete(bodyweight: 78 kg, goal: bulk) {

    workout "Push Day" {
        exercise "Bench Press"    { sets: 4, reps: 6, weight: 80 kg }
        exercise "Overhead Press" { sets: 3, reps: 8, weight: 50 kg }
        progress weight 2.5 kg/week
    }

    meal "Breakfast" {
        protein: 40 g
        carbs:   80 g
        fat:     15 g
    }

    target protein >= 1.8 g per kg of bodyweight
    goal   lose 0.5 kg/week

    plan week with workouts: ["Push Day"], meals: ["Breakfast"]
}
```

**Justification (Sebesta §1.3 evaluation criteria).**
FitLang is designed to optimise **writability** for its application domain (Sebesta §1.3.2 — writability is always discussed *relative to a domain*) and **reliability** through a strict, unit-aware type system (§1.3.3). The domain user — a lifter planning a training block, not a software engineer — should be able to express a workout in roughly the same words they would write in a notebook (`sets: 4, reps: 6, weight: 80 kg`); this is the form-and-meaning correspondence Sebesta lists under readability (§1.3.1) and the closeness-to-application-domain factor that drives the *cost of writing programs* (§1.3.4). Reliability is bought by typed physical quantities that disallow arithmetic across dimension families (`80 kg + 1 m` is a static type error) and disallow same-unit multiplication or division (`kg * kg`, `kg / kg`); this is the same trade-off Sebesta credits Java for taking against C in §1.6, where mandatory checks cost execution time but eliminate a class of bugs entirely. The criteria FitLang knowingly sacrifices are **generality** (it cannot, and should not, be used outside fitness/nutrition planning — Sebesta lists generality only as a secondary criterion, §1.3) and **cost of execution** (a tree-walking Java interpreter is at least an order of magnitude slower than compiled code per §1.7's discussion of pure interpretation, but a one-second plan calculation is irrelevant for the domain).

## 4.2 Lexical Structure **[P1]**

### 4.2.1 Character set, line endings, whitespace

FitLang source is UTF-8. The lexer treats the source as a stream of Unicode code points, but the only places non-ASCII code points may appear are inside string literals and inside line comments — every other lexeme is ASCII. Whitespace (`U+0020` space, `U+0009` tab, `U+000A` LF, `U+000D` CR) is insignificant: it separates tokens but carries no other meaning. Indentation is not part of the grammar; we explicitly reject Python-style significant indentation (Sebesta §3.1) because the block syntax already uses `{ }` and double-duty would only cost readability.

### 4.2.2 Comments

A `#` outside a string literal starts a line comment that runs to the next newline. There is no block-comment form. One comment style keeps the lexer's comment automaton trivial and matches the configuration-language feel of the surface syntax.

```fitlang
# Cutting block, week 3
workout "Push Day" { ... }
```

### 4.2.3 Token classes

Every lexeme produced by the scanner falls into exactly one of the classes below. Class names in **bold** are the terminal symbols used by §4.3's EBNF.

| Class | Regex (POSIX-extended) | Examples |
|---|---|---|
| **RESERVED** | one of the listed reserved words (table 4.2.5) | `athlete`, `workout`, `let` |
| **MODE_LIT** | one of the listed mode literals (table 4.2.5) | `bulk`, `cut`, `lose` |
| **UNIT** | one of the listed unit lexemes (table 4.2.6) | `kg`, `kcal`, `week` |
| **NUM_LIT** | `[0-9]+ (\.[0-9]+)?` | `80`, `2.5`, `0.5` |
| **STRING_LIT** | `"` *( any Unicode except `"` and newline; `\\` and `\"` escapes )* `"` | `"Push Day"`, `"Kahvaltı"` |
| **IDENT** | `[A-Za-z_][A-Za-z0-9_]*` minus any RESERVED / MODE_LIT / UNIT lexeme | `bodyweightTarget`, `dailyKcal` |
| **PUNCT** | one of `{ } [ ] ( ) , : / + - * = > < >= <= == !=` | `{`, `>=`, `:` |

Numeric literals carry no sign — leading `-` is the unary minus operator, lexed as the `-` punctuation token. Exponent form (`1.5e3`) is rejected; the domain has no quantity that benefits from it. There is no integer / float distinction at the lexical level — both `80` and `80.0` produce a single **NUM_LIT** token whose attribute is the numeric value; the type system (§4.5) decides where each is admissible.

### 4.2.4 String literals

A string literal is delimited by ASCII double quotes. Inside the literal any Unicode code point is permitted except an unescaped `"` or raw newline (literal newlines in source are rejected to keep error recovery line-local). Three escape sequences are recognized: `\"` (literal quote), `\\` (literal backslash), and `\n` (line feed). Octal, hex, and `\u` escapes are not supported — labels only need a way to embed quotes, backslashes, and the occasional line break.

### 4.2.5 Reserved words and mode literals

FitLang uses a **closed lexicon**: every unquoted alphabetic lexeme is either a reserved word, a mode literal, or a unit. The user introduces names only via string literals (workout / exercise / meal labels) or via numeric `let` bindings (one **IDENT** class, restricted by §4.5 to scalar and quantity types). This is the "fully reserved" stance Sebesta credits to Pascal in §3.1 — a reserved word can never be redeclared as an identifier, eliminating the keyword-vs-identifier ambiguity ALGOL 60 left to context.

Reserved words (case-sensitive):

```
athlete  workout  exercise  meal  progress  target  goal  plan  week
with     workouts meals     let   bodyweight
sets     reps     weight    protein  carbs   fat
per      of       and
when     else     routine   use
```

The last row introduces the §2 control structure (`when` / `else`) and the §2 user-defined function template (`routine` declaration, `use` call). They participate in the closed lexicon on the same terms as every other reserved word — no shadowing, no use as `IDENT`.

Mode literals (case-sensitive — finite enumerations consumed positionally in §4.3):

```
bulk  cut  maintain         # goal modes
lose  gain                   # weight-change directions
```

Mode literals are syntactically distinct from reserved words because §4.3 admits them only in fixed positions (`goal: <mode>`, `<direction> <quantity>`); flagging them with their own token class catches misuse at parse time rather than via a semantic check.

### 4.2.6 Units

Units are recognized as their own token class so that the parser can require a **NUM_LIT** **UNIT** pair anywhere a quantity is expected (per the §1 design call: lex two tokens, pair at parse time). Each unit lexeme carries an attribute identifying its dimension family — Mass, Energy, Time — used by §4.5's coercion rules.

| Family | Lexemes |
|---|---|
| Mass | `g`, `kg`, `lb` |
| Energy | `kcal`, `kJ` |
| Time | `s`, `min`, `h`, `day`, `week` |

Macronutrients (protein, carbs, fat) are themselves masses — `protein: 80 g` and `bodyweight: 78 kg` carry quantities of the same family, and within-family coercion (§4.5) means `bodyweight: 70000 g` is legal and equal to `bodyweight: 70 kg`. Three families instead of four keeps the type rule simple and lets the cross-family error rule (`80 kg + 200 kcal` is a static type error) carry its weight without artificial splits. `week` is both a reserved word (in `plan week`) and a unit (in `2.5 kg/week`); the scanner emits the same lexeme and §4.3 distinguishes by position. Compound rate forms like `kg/week` are *not* a single lexeme — the scanner emits three tokens (`NUM_LIT` `UNIT` `/` `UNIT`) and the parser builds the rate.

### 4.2.7 Punctuation

The complete punctuation alphabet is `{`, `}`, `[`, `]`, `(`, `)`, `,`, `:`, `/`, `=`, `>`, `<`, `>=`, `<=`, `==`, `!=`, `+`, `-`, `*`. Maximal-munch (Sebesta §4.2 longest-match rule) disambiguates `>=`/`<=`/`==`/`!=` against their single-character prefixes; the scanner reads as many characters as form a valid punctuation lexeme before emitting a token. Equality (`==`) and inequality (`!=`) are reserved here at the lexical level; their semantics — operand types, whether they compare quantities by canonicalized value or by surface unit — are specified in §4.7 (Part 2). There is no `===`: FitLang has no reference-typed values, so the JavaScript-style value-vs-identity distinction would be ceremony with no payoff. There is no `;` — newlines and `,` carry list and statement separation.

### 4.2.8 Identifiers

**IDENT** is `[A-Za-z_][A-Za-z0-9_]*` minus the closed lexicon of §4.2.5–4.2.6. ASCII only — Unicode identifiers add a normalization burden (NFC vs NFD homoglyphs) Sebesta calls out in §3.1, and the DSL has no payoff. Case-sensitive: `bodyweight` (the reserved word) and `Bodyweight` (a legal **IDENT**) are distinct lexemes; we follow the C/Java convention and reject Fortran-style case folding (Sebesta §3.1 lists case-insensitivity as a historical outlier). The scanner enforces the "minus reserved" rule by checking each candidate identifier against the reserved/mode/unit tables before emitting **IDENT**, so the parser never has to disambiguate.

## 4.3 Syntax (EBNF) **[P1]**

### 4.3.1 Notation

The grammar is written in the extended BNF dialect Sebesta uses in §3.3: `→` defines a nonterminal, `|` separates alternatives, `{ X }` denotes zero-or-more repetition, `[ X ]` denotes an optional element. Nonterminals are written in `<angle-brackets>`. Terminals are either literal lexemes in `"double quotes"` (matching the surface syntax verbatim) or the uppercase token-class names introduced in §4.2.3 (`NUM_LIT`, `STRING_LIT`, `IDENT`, `UNIT`). The start symbol is `<program>`.

### 4.3.2 Productions

```ebnf
(* — top level — *)
<program>          → <athlete-block>

<athlete-block>    → "athlete" "(" <athlete-params> ")"
                     "{"
                         { <routine-decl> }
                         <body-sections>
                         <rules-section>
                         <schedule-stmt>
                     "}"

<athlete-params>   → "bodyweight" ":" <quantity> "," "goal" ":" <goal-mode>
<goal-mode>        → "bulk" | "cut" | "maintain"

(* — at least one of plan / intake; both allowed in fixed order — *)
<body-sections>    → <plan-section> [ <intake-section> ]
                   | <intake-section>

(* — training plan section: one or more workouts — *)
<plan-section>     → <workout-decl> { <workout-decl> }

<workout-decl>     → "workout" STRING_LIT
                     "{"
                         <workout-item> { <workout-item> }
                         [ <progress-stmt> ]
                     "}"

(* — a workout body is built from inline exercise records or
     calls to a previously-declared routine template — *)
<workout-item>     → <exercise-decl> | <routine-call>

(* — <exercise-decl> is the §2 structured type (record schema):
     a fixed set of named, typed fields evaluated positionally — *)
<exercise-decl>    → "exercise" STRING_LIT
                     "{"
                         "sets"   ":" NUM_LIT ","
                         "reps"   ":" NUM_LIT ","
                         "weight" ":" <quantity>
                     "}"

<progress-stmt>    → "progress" "weight" <rate>

(* — user-defined parameterised template: a named routine,
     declared once at athlete scope, instantiated at any
     workout body via <routine-call>.  Satisfies §2's
     "user-defined functions with parameters" requirement
     in a declarative DSL idiom — the routine is not a
     procedure but a parameterised exercise group. — *)
<routine-decl>     → "routine" STRING_LIT "(" <routine-params> ")"
                     "{"
                         <exercise-decl> { <exercise-decl> }
                         [ <progress-stmt> ]
                     "}"
<routine-params>   → IDENT ":" UNIT { "," IDENT ":" UNIT }
<routine-call>     → "use" STRING_LIT "(" <quantity> { "," <quantity> } ")"

(* — intake section: one or more meals — *)
<intake-section>   → <meal-decl> { <meal-decl> }

<meal-decl>        → "meal" STRING_LIT
                     "{"
                         <macro-decl> { <macro-decl> }
                     "}"

<macro-decl>       → <macro-name> ":" <quantity>
<macro-name>       → "protein" | "carbs" | "fat"

(* — rules section: zero or more nutritional/progression rules — *)
<rules-section>    → { <rule-decl> }
<rule-decl>        → <target-decl> | <goal-rate-decl> | <let-decl> | <when-decl>

<target-decl>      → "target" <macro-name> <rel-op> <quantity>
                         [ "per" UNIT "of" "bodyweight" ]
<goal-rate-decl>   → "goal" <direction> <rate>
<direction>        → "lose" | "gain"
<let-decl>         → "let" IDENT "=" <expression>

(* — control structure (§2 requirement): rules inside the
     "then" block apply only when the athlete's declared
     goal mode matches; the optional "else" branch carries
     the rules that apply otherwise.  The dispatch value
     is the goal mode bound in <athlete-params>, so the
     conditional is fully decidable at compile time
     (a static <if> in Sebesta §8.1 terms) — there is no
     run-time predicate. — *)
<when-decl>        → "when" "goal" ":" <goal-mode>
                         "{" { <rule-decl> } "}"
                         [ "else" "{" { <rule-decl> } "}" ]

(* — schedule: exactly one, closes the program;
     at least one of workouts / meals must appear — *)
<schedule-stmt>    → "plan" [ NUM_LIT ] "week" "with" <schedule-clauses>
<schedule-clauses> → "workouts" ":" <name-list> [ "," "meals" ":" <name-list> ]
                   | "meals"    ":" <name-list>
<name-list>        → "[" STRING_LIT { "," STRING_LIT } "]"

(* — quantities, rates, expressions — *)
<quantity>         → NUM_LIT UNIT
<rate>             → NUM_LIT UNIT "/" UNIT

(* P1 expression layer: additive operators only.  Multiplicative
   operators ("*", "/" outside of <rate>), parenthesised grouping,
   and the full precedence/associativity table are added in §4.7
   (Part 2).  "+" and "-" are left-associative; with no other
   operators present in P1 there is no precedence question to
   answer here.  Operand-type rules (only same-family quantities
   may be added; cross-family is a static type error) live in §4.5. *)
<expression>       → <term> { <add-op> <term> }
<term>             → <quantity> | NUM_LIT | IDENT
<add-op>           → "+" | "-"

<rel-op>           → ">=" | "<=" | ">" | "<" | "==" | "!="
```

### 4.3.3 What the start symbol enforces

The grammar is *structured* in the Pascal sense (Sebesta §3.3, Pascal program template): a FitLang program is not a flat list of declarations but a fixed skeleton of sections in a required order — **header → plan → intake → rules → schedule**. Three consequences follow at parse time, before any semantic analysis runs:

1. **Every program has exactly one athlete context.** `<program> → <athlete-block>` admits no alternative; a file containing two `athlete(...)` blocks or none fails the LL(1) prediction set on the first token. This is the rule the §4.5 type system relies on when it resolves `bodyweight` without a scope-lookup step.
2. **An empty body, or a body without a schedule, is unparseable.** `<body-sections>` requires at least one of plan / intake — written as a two-alternative production rather than `[ <plan-section> ] [ <intake-section> ]`, which would admit the empty derivation. Each section that *is* present must be non-empty (one-or-more, written `<workout-decl> { <workout-decl> }`, never `{ <workout-decl> }`). `<schedule-stmt>` is mandatory and is the last child of `<athlete-block>`. The parser therefore rejects three classes of malformed program with one token of look-ahead and a domain-specific message: empty athlete blocks, sections declared but empty (e.g., `meal "Breakfast" {}`), and programs that never schedule themselves. Workout-only and meal-only programs *are* legal — a diet-only client and a strength-only client are both first-class FitLang programs.
3. **Rules reference an existing goal mode.** `<athlete-params>` fixes the goal mode (`bulk` / `cut` / `maintain`) before any `<rule-decl>` is parsed, so `<goal-rate-decl>` in §4.4's semantics can be checked for consistency with the header (e.g., `goal: bulk` + `goal lose 0.5 kg/week` is a static contradiction) without any forward reference. The `<when-decl>` control structure piggy-backs on the same property: because the goal mode is fixed before any rule is parsed, the `when goal: <mode>` dispatch is decidable at compile time and the unused branch can be eliminated before §4.4's semantic phase even runs.
4. **Routines are declared before they are used.** `{ <routine-decl> }` appears at the head of `<athlete-block>`, strictly before `<body-sections>`. A `<routine-call>` inside a `<workout-decl>` therefore always refers to a routine name introduced earlier in the same block — the same no-forward-reference discipline §4.6.3 imposes on `let`. The grammar makes this a syntactic property; semantics need only check argument arity and unit families against the matching `<routine-decl>`.

This is the originality argument restated as a grammar property: a general-purpose language (C, Python) cannot bake its program shape into the start symbol because it does not know what a program is *for*; FitLang does, so it should.

### 4.3.4 Lexical / syntactic boundary

Two design calls are visible in the productions above and worth flagging because they push work across the lex/parse boundary:

- **Quantities are pairs at the parse level, not at the lex level.** `<quantity> → NUM_LIT UNIT` follows the §4.2.6 decision: the scanner emits two tokens, the parser fuses them. The alternative — a single `QUANTITY_LIT` lexeme matching `[0-9]+(\.[0-9]+)?[a-zA-Z]+` — would force the scanner to know the unit family table and would re-couple lexical and semantic concerns Ada deliberately separates.
- **Compound rates (`kg/week`) are three tokens, not one.** `<rate> → NUM_LIT UNIT "/" UNIT` reuses the existing `/` punctuation rather than introducing a `RATE_LIT` lexeme. The parser distinguishes division-of-quantities (a §4.7 expression form, P2) from rate construction by position: a `/` immediately following a `NUM_LIT UNIT` pair inside a `<progress-stmt>` or `<goal-rate-decl>` constructs a rate, anywhere else it is the binary operator.

### 4.3.5 Parsing class

The grammar is **LL(1)** as written: every alternative inside `<rule-decl>` and `<athlete-block>`'s body is selected by a unique reserved word in the FIRST set (`workout`, `meal`, `target`, `goal`, `let`, `when`, `progress`, `plan`), and the one-or-more constructs (`<plan-section>`, `<intake-section>`) commit on their distinguishing first token (`workout`, `meal`). The two alternative productions added to support diet-only and workout-only programs — `<body-sections>` and `<schedule-clauses>` — are also LL(1) by inspection: `FIRST(<plan-section>) = { workout }` is disjoint from `FIRST(<intake-section>) = { meal }`, and the two alternatives of `<schedule-clauses>` are selected by `workouts` vs `meals`. No left recursion appears; the only repetition is right-recursive in EBNF form. This matches the recursive-descent implementation strategy planned for D2 and the parsing-class discussion Sebesta gives in §4.4 — LL(1) is the cheapest grammar class that buys us readable hand-written error messages and predictable single-pass parsing.

The §2 extensions preserve LL(1) by construction. Each new alternative is selected by a reserved word disjoint from every existing FIRST set:

- `FIRST(<routine-decl>) = { routine }`, disjoint from `FIRST(<plan-section>) = { workout }` and `FIRST(<intake-section>) = { meal }`. The optional `{ <routine-decl> }` prefix in `<athlete-block>` therefore commits or skips on the first token after `{` with a single look-ahead.
- Inside a workout body, `FIRST(<exercise-decl>) = { exercise }` and `FIRST(<routine-call>) = { use }` are disjoint, so `<workout-item>` is LL(1).
- `FIRST(<when-decl>) = { when }` is disjoint from the existing `<rule-decl>` alternatives' FIRSTs (`{ target, goal, let }`), so adding it as a fourth alternative is direct. The optional `else`-branch is selected by `{ else }`, which does not appear in the FOLLOW set of `<when-decl>`'s body (the body is closed by `}`, after which only another `<rule-decl>` start, `plan`, or the `else` look-ahead can follow — and the closing `}` consumes the choice deterministically before `else` is offered).
- `FIRST(<routine-params>) = { IDENT }` is the only IDENT-headed production in `<athlete-block>`, but it appears only inside a `(` ... `)` pair already committed by the preceding `routine STRING_LIT (`, so no ambiguity arises with `<expression>`'s `<term> → IDENT` alternative (which is reachable only through `<let-decl>`).

### 4.3.6 Sample derivation

For the §4.1 sample program, the leftmost derivation begins:

```
<program>
⇒ <athlete-block>
⇒ "athlete" "(" <athlete-params> ")" "{" <plan-section> <intake-section> <rules-section> <schedule-stmt> "}"
⇒ "athlete" "(" "bodyweight" ":" <quantity> "," "goal" ":" <goal-mode> ")" "{" ...
⇒ "athlete" "(" "bodyweight" ":" NUM_LIT UNIT "," "goal" ":" "bulk" ")" "{" ...
⇒ "athlete" "(" "bodyweight" ":" 78 kg "," "goal" ":" "bulk" ")" "{" ...
```

A full parse tree is given in D3's worked example (Part 1 deliverable, three valid programs). The point of the partial derivation here is to show that the first six tokens of any FitLang program are forced by the grammar — `athlete ( bodyweight : NUM_LIT UNIT` — which is what gives D2's parser its strong front-end error recovery story.

## 4.4 Semantics **[P2]**

### 4.4.1 Style and meta-language

The semantics is given **operationally** in big-step (natural-semantics) style — the form Sebesta surveys in §3.5.1 and credits to Plotkin/Kahn. A FitLang program denotes a value of type `WeeklyPlan` (defined in §4.4.2); each evaluation rule has the shape `env ⊢ node ⇓ value`, read "in environment *env*, AST node *node* evaluates to *value*." Declarations use a separate judgement `env ⊢ decl ⇝ env'`, read "in environment *env*, declaration *decl* extends the environment to *env'*" — used by the (LET) rule of §4.4.5. We pick operational over denotational (§3.5.2) and axiomatic (§3.5.3) for the same reason every Sebesta-style intro DSL does: it maps directly onto the tree-walking interpreter in D2 and is the form the user can defend in the quiz without a domain-theory aside.

A FitLang program is **pure, declarative, and one-shot**: no input, no I/O during evaluation, no mutation, no nondeterminism. Evaluation order is fixed by the grammar (top-down, left-to-right over the children of each node) but the order is observationally irrelevant — by §4.6.6 immutability, two visit orders that respect the no-forward-reference rule of §4.6.3 produce the same `WeeklyPlan`.

### 4.4.2 Program meaning — the `Plan` value

The denotation of any well-typed FitLang program is a single `Plan` record — an athlete header plus an ordered list of `Week`s, one per week in the scheduled horizon (§4.4.8):

```
Plan = {
    athlete  : { bodyweight : Mass, goal : GoalMode, goalRate : Maybe Rate(Mass, Time) },
    weeks    : [ Week ]                          // length N from <schedule-stmt>, N ≥ 1
}

Week = {
    index               : Number,                // 1..N
    projectedBodyweight : Mass,                  // bodyweight + goalRate × (index − 1 week)
    workouts            : [ EvaluatedWorkout ],  // schedule order
    meals               : [ EvaluatedMeal ],     // schedule order
    targets             : [ TargetReport ]       // re-evaluated against this week's bodyweight
}

EvaluatedWorkout = {
    label    : String,
    exercises: [ { label, sets, reps, weight : Mass } ]   // weight is per-week, see §4.4.6
}

EvaluatedMeal     = { label : String, protein : Mass, carbs : Mass, fat : Mass, kcal : Energy }
TargetReport      = { macro : MacroName, op : RelOp, threshold : Mass, planned : Mass, passes : Bool }
```

A program with `plan week ...` (no integer) degenerates to `N = 1` and a single-element `weeks` list — the "snapshot" reading. A program with `plan 8 week ...` produces an 8-week training block where each week's bodyweight, exercise weights, and target thresholds are recomputed under the rates declared in the program. Nothing else is observable — no side effects, no random state — which is why §4.4.1's "operational, pure, one-shot" framing is enough to characterise meaning.

### 4.4.3 Atomic forms

```
env ⊢ NUM_LIT n            ⇓ n  : Number
env ⊢ NUM_LIT n  UNIT u    ⇓ canonicalize(n, u)  : Quantity (family u)
env ⊢ NUM_LIT n  UNIT u₁ "/" UNIT u₂
                            ⇓ canonicalize(n, u₁) / canonicalize(1, u₂)  : Rate(family u₁, family u₂)
env ⊢ IDENT x              ⇓ env(x)
```

`canonicalize(n, u)` rescales `n` to the canonical unit of `family u` per §4.5.4 — `78 kg` becomes `78000 g`, `200 kcal` stays `200 kcal`, `1 week` becomes `604800 s`. All later arithmetic in the interpreter is on canonicalized values; the printer reinflates to the surface unit a human would read (kg, kcal, week).

### 4.4.4 Expressions

The expression rules mirror §4.5.8 and add the value-level operation:

```
env ⊢ e₁ ⇓ v₁    env ⊢ e₂ ⇓ v₂    v₁ ⊕ v₂ defined per §4.5
————————————————————————————————————————————————————————————
env ⊢ e₁ "⊕" e₂  ⇓ v₁ ⊕ v₂
```

The `defined per §4.5` premise is discharged statically by the type checker; at run time the interpreter never encounters an undefined arithmetic case, so there is no need for a run-time "operands must agree" guard. Unit canonicalization makes `+` and `-` ordinary arithmetic on canonicalized numbers, `*` and `/` ordinary scalar arithmetic (§4.5.4 rules 3–4) and `Rate × Time → Quantity` an ordinary multiply followed by a family relabel (§4.5.5).

### 4.4.5 Declarations and the environment

A single linear pass over the body of `<athlete-block>` threads an environment of let-bindings:

```
            env ⊢ e ⇓ v
————————————————————————————————    (LET)
env ⊢ "let" IDENT x "=" e   ⇝   env ∪ { x ↦ v }
```

`⇝` is the "extends-environment" relation — `<let-decl>`s are not values, they are *bindings*. By §4.6.3 no-shadowing rule, `x ∉ dom(env)` is required; the type checker has already ensured this, so the rule is unconditional at run time.

`<exercise-decl>`, `<meal-decl>`, `<workout-decl>`, `<routine-decl>` are declarations whose effect is to add an entry to the appropriate label space (workout / exercise / meal / routine). They do not contribute to the let-environment.

### 4.4.6 Workouts and exercises

A `<workout-decl>` is evaluated **once per week** in the scheduled horizon — the same declaration yields a different `EvaluatedWorkout` in week 1 vs week 4 because `<progress-stmt>` injects a per-week offset into each exercise's weight. Without a `<progress-stmt>` the weekly evaluations are identical and the per-week duplication is observationally a no-op.

```
env ⊢ q ⇓ w₀ : Mass    progressRate(workout) = r : Rate(Mass, Time) ∪ { ⊥ }
————————————————————————————————————————————————————————————————————   (EXERCISE @ week k)
env ⊢ exercise L { sets: s, reps: r_s, weight: q }
              ⇓_k  { label=L, sets=s, reps=r_s, weight = w₀ + r × ((k − 1) week) }
```

where `r × ((k−1) week)` is `0 kg` when `r = ⊥` (no `<progress-stmt>` in the enclosing workout) or when `k = 1`. This is the operational use of the `Rate × Time → Quantity` rule of §4.5.5.

A `<progress-stmt>` is not itself a value — its sole effect is to bind `progressRate(workout)` for the rule above. The rate is shared across every exercise in the same workout (one `<progress-stmt>` per `<workout-decl>` by §4.3); a workout without one has every week identical.

### 4.4.7 Routines and routine calls

A `<routine-decl>` `routine "R" (p₁: u₁, …, pₙ: uₙ) { body }` is recorded in the routine label space as a thunk of `(params, body)`. It is not evaluated until used.

```
env ⊢ a₁ ⇓ v₁ : Quantity (family u₁)    ...    env ⊢ aₙ ⇓ vₙ : Quantity (family uₙ)
env' = env ∪ { p₁ ↦ v₁, ..., pₙ ↦ vₙ }
env' ⊢ body  ⇓  [ EvaluatedExercise, ... ]
————————————————————————————————————————————————————————————————————   (USE)
env ⊢ use "R" (a₁, ..., aₙ)   ⇓   [ EvaluatedExercise, ... ]
```

Argument types are checked statically against the routine's `UNIT`-annotated parameter declarations. The routine's body is evaluated in a *fresh* environment that extends `env` only with the parameter bindings — routine bodies cannot see the caller's `let`-bindings, only the athlete-scope ones present at the routine's *declaration* site. This is static (lexical) scoping in Sebesta §5.5.1 form, applied to a parameterized declarative template.

### 4.4.8 The `plan N week` schedule

`<schedule-stmt>` is the single output-producing construct. It picks a horizon `N ≥ 1` (default 1 when the optional `NUM_LIT` is omitted), selects from the declared workouts and meals by label, and assembles one `Week` per index — each week re-evaluating workouts under their progression rates and targets under the projected bodyweight.

```
N        = NUM_LIT in <schedule-stmt>, or 1 if absent
labels_w = [ L₁, L₂, ... ]    labels_m = [ M₁, M₂, ... ]
G        = the rate from the unique <goal-rate-decl>, or ⊥
bw₀      = env.bodyweight

  for each k ∈ 1..N:
      bw_k = bw₀ + G × ((k − 1) week)                        // bw₀ if G = ⊥
      W_k  = [ evalWorkout(Lᵢ) at week k        for Lᵢ in labels_w ]   // §4.4.6
      M_k  = [ evalMeal(Mⱼ)                      for Mⱼ in labels_m ]   // meals constant across weeks
      T_k  = [ evalTarget(t, M_k, bw_k)          for t in targets_in_scope ]   // §4.4.9
      Week_k = { index=k, projectedBodyweight=bw_k, workouts=W_k, meals=M_k, targets=T_k }
————————————————————————————————————————————————————————————————————————   (PLAN)
env ⊢ plan [N] week with workouts: labels_w, meals: labels_m
        ⇓ Plan { athlete = { bodyweight=bw₀, goal, goalRate=G }, weeks = [Week_1, ..., Week_N] }
```

"declared" is decided at parse time (§4.6.3 forward-reference rule); a label in the schedule that names no declaration is a static error, not a run-time one. The list order in each week follows the **schedule order** (the order the user wrote them in `[ ... ]`), not the declaration order — this gives the lifter direct control over Monday-through-Sunday ordering without introducing a separate day construct.

Meals are constant across weeks because `<meal-decl>` has no rate construct — a lifter changes intake by writing a new meal declaration (and re-scheduling), not by attaching a progression to the existing one. This is intentional: bodyweight, lift load, and macro targets all move; the *meals you eat* are discrete plans the lifter writes. If a future version wants per-week intake changes, the natural extension is `<when-decl>` keyed on `week index` rather than a meal-side rate.

### 4.4.9 Targets and macro arithmetic

A `<target-decl>` is evaluated **once per week** against that week's scheduled meals and projected bodyweight (not against every declared meal, and not once for the whole plan):

```
env ⊢ q ⇓ t : Mass
planned_k = Σ { meal.<macro> | meal ∈ scheduled meals at week k }
————————————————————————————————————————————————————————————————   (TARGET, absolute form @ week k)
env ⊢ target <macro> <op> q
            ⇓_k TargetReport { macro, op, threshold=t, planned=planned_k, passes=(planned_k <op> t) }
```

The "per kg of bodyweight" form rescales the threshold by the projected bodyweight of the current week:

```
env ⊢ q ⇓ t : Mass    bw_k = projected bodyweight at week k (§4.4.8)
————————————————————————————————————————————————————————————————   (TARGET, per-bw form @ week k)
env ⊢ target <macro> <op> q per kg of bodyweight
            ⇓_k TargetReport { macro, op, threshold = t × (bw_k in kg), planned=planned_k, passes }
```

The "× bw" here is the one place the language uses *quantity scaled by a scalar drawn from a quantity's numeric value* — §4.5.4 forbids `Mass × Mass`, but the `per UNIT of bodyweight` form is sugar for "treat the projected bodyweight in the named unit as a dimensionless multiplier." This is the same trick Ada `'Pos` does for enumerations (Sebesta §6.4) — a privileged projection from a typed value to a number, available only in the named sugar form, never as a general expression operator.

`<goal-rate-decl>` drives the per-week bodyweight projection of §4.4.8 and is consistency-checked against the header `goal` mode (`bulk` with `lose` is rejected) by the type checker. Meals are constant across weeks (§4.4.8), so `planned_k` differs across weeks only when a future extension lets the schedule pick different meals per week.

### 4.4.10 The `when` control structure

`<when-decl>` is **decided at compile time** — the dispatch value is the goal mode bound in `<athlete-params>`, which is a literal in the header (§4.3.3 rule 3). A pre-semantic pass eliminates the dead branch:

```
goal-mode in env  =  M
————————————————————————————————   (WHEN-then)
env ⊢ when goal: M { rules₁ } else { rules₂ }   ↦   rules₁

goal-mode in env  ≠  M
————————————————————————————————   (WHEN-else)
env ⊢ when goal: M { rules₁ } else { rules₂ }   ↦   rules₂   (or ∅ if no else)
```

The surviving rule list is spliced into the enclosing `<rules-section>` and evaluated by the rules above. This is Sebesta §8.1's *static if* — an alternative chosen at compile time — and is the reason `when` is in the language at all: it lets the user write goal-mode-specific intake or targets without paying any run-time dispatch cost.

### 4.4.11 Program rule and worked example

```
env₀ = ∅
env₁ = env₀ ∪ { bodyweight ↦ ..., goal ↦ ... }    // from <athlete-params>
env₂ = thread <let-decl>s through env₁              // §4.4.5
Plan = evaluate <schedule-stmt> in env₂             // §4.4.8 (N ≥ 1 weeks)
————————————————————————————————————————————————   (PROGRAM)
⊢ <athlete-block>  ⇓  Plan
```

For a variant of the §4.1 sample (bodyweight 78 kg, goal bulk, `"Push Day"` with bench at 80 kg and `progress weight 2.5 kg/week`, `"Breakfast"` at protein 40 g, `target protein >= 1.8 g per kg of bodyweight`, `goal gain 0.25 kg/week`, scheduled as `plan 4 week with workouts: ["Push Day"], meals: ["Breakfast"]`):

1. (LET) — none; env after header = `{ bodyweight ↦ 78 kg, goal ↦ bulk, goalRate ↦ +0.25 kg/week }`.
2. (EXERCISE / PROGRESS) — `"Push Day"` records base weights and `progressRate = 2.5 kg/week`.
3. (PLAN) — N = 4. For each k in 1..4:
   - `bw_k = 78 kg + 0.25 kg × (k−1)` → 78, 78.25, 78.5, 78.75 kg.
   - Bench `weight_k = 80 kg + 2.5 kg × (k−1)` → 80, 82.5, 85, 87.5 kg.
   - `threshold_k = 1.8 g × bw_k_in_kg` → 140.4, 140.85, 141.3, 141.75 g.
   - `planned_k = 40 g` (constant); `passes_k = false` in every week.
4. The `Plan` printed by the interpreter is a four-week table with the trajectory above.

D3's three valid programs each include their final printed `Plan` as the "executed output" the handout asks for; at least one of them uses `N > 1` so the per-week projection is exercised.

## 4.5 Type System **[P2]**

### 4.5.1 Type universe

FitLang has a small, closed type universe — there is no syntax for the user to introduce a new type, only to use the ones the language gives:

| Category | Types |
|---|---|
| **Scalar** | `Number` |
| **Quantity** (dimensional family + canonical unit) | `Mass` (canonical `g`), `Energy` (canonical `kcal`), `Time` (canonical `s`) |
| **Rate** (quotient of two quantities) | `Rate(D₁, D₂)` for any two families D₁, D₂ — in practice only `Rate(Mass, Time)` (`kg/week`) appears in P1 idiom |
| **Mode** (finite enumeration) | `GoalMode = { bulk, cut, maintain }`, `Direction = { lose, gain }` |
| **Label** | `String` (only as the head of a `<workout-decl>` / `<exercise-decl>` / `<meal-decl>` / `<routine-decl>`; not first-class — no string operations) |
| **Record kind** | `Exercise`, `Meal`, `Workout`, `Routine` — each a fixed schema named by its kind keyword (§4.5.6) |

Every `IDENT` introduced by `<let-decl>` has one of the three RHS-admissible types — `Number`, `Quantity D`, or `Rate(D₁, D₂)` — fixed at its binding site. Labels and record kinds are not nameable through `IDENT`; they live in the string-label space of §4.6.1 and the parser-tagged AST node space respectively.

### 4.5.2 Strong typing stance

FitLang is **strongly typed** in Sebesta §6.12's sense: every type error is caught — at compile time in our case — and no operation silently coerces across the dimensional families of §4.5.4. This is the same trade-off Sebesta credits Java for taking against C in §1.6: we pay a small ergonomic cost (the user occasionally annotates a unit they "obviously" meant) and in return remove a whole class of plan-corrupting bugs — a meal whose `protein: 80 kcal` slipped past, a `progress weight 2.5 kcal/week` that would have shifted a bulk into nonsense.

The closed type universe of §4.5.1 is what makes strong typing affordable: with no user-defined types, no inheritance, no generics beyond the parametric `Rate(D₁, D₂)`, the type checker is a single linear pass over the AST with O(declarations) work — the property Sebesta lists in §6.12 as the practical price of strong typing in ML/Ada and the practical reason C ducks it.

### 4.5.3 Type equivalence — name, not structural

Two values have the same type iff their type expressions are spelled the same in §4.5.1's universe. Concretely:

- Two `Quantity` values are the same type iff they belong to the same dimensional family. `78 kg` and `200 g` are both `Mass`; `200 g` and `200 kcal` are not the same type.
- Two record values are the same type iff they were built from the same **kind keyword**. An `exercise` block and a `meal` block are never the same type, even if they had identical field names (they can't, by grammar). Two `exercise` blocks are always the same type — the grammar admits exactly one `exercise` schema, so the question of "do their fields match" is decided in advance by the parser.

This is **name equivalence** (Sebesta §6.14) — Ada and classical Pascal use it for record types; we take it for the same reason: the kind keyword is the carrier of identity, and two records with the same shape but different kinds (a future `exercise` vs a future `circuit`) should be different types by definition, not by coincidence of field layout. The alternative, structural equivalence (Sebesta §6.14 — ML, Modula-3, Go interfaces, TypeScript), buys flexibility FitLang has no use for: the grammar already pins each kind to one schema, so structural and name equivalence agree on every program FitLang can parse. We pick name equivalence because it gives a one-sentence rule (*"same kind ⇒ same type, different kind ⇒ different type"*) and because it scales cleanly if a later version adds a second exercise-like kind without forcing it to interoperate with the original.

E2 (the AI-vs-human equivalence experiment in D4) revisits this decision against Sebesta §6.14 directly.

### 4.5.4 Dimensional families and within-family coercion

Quantities live in three disjoint **dimensional families** (Mass, Energy, Time — §4.2.6). The type checker rules:

1. **Within a family, units coerce to the canonical unit at the value level.** `78 kg` and `200 g` are both `Mass`; the checker treats them as equal-typed and the interpreter (§4.4) canonicalizes both to grams before any arithmetic. `78 kg + 200 g` is a legal `Mass` value (= 78 200 g); `1 lb + 1 kg` is a legal `Mass` value (≈ 1453.6 g). This is the F#/Frink approach to dimensional values and the rule Ada `with Unchecked_Conversion` lets you opt out of (Sebesta §6.5).
2. **Across families, every operation is a static type error.** `80 kg + 200 kcal` does not type-check. `80 kg * 2 kg` does not type-check (same-family multiplication would produce `Mass²`, which has no inhabitant in FitLang's universe — see §4.5.5). `80 kg / 80 kg` does not type-check either (would produce dimensionless `Number`, again outside the universe). These are *deliberate omissions*, not oversights: the domain has no use for square-mass or dimensionless ratios, and admitting them would force the universe open.
3. **Scalar × quantity is legal and yields a quantity of the same family.** `2 * 80 kg` is `Mass` (= 160 kg). This is the only multiplication form the checker accepts.
4. **Quantity / scalar is legal and yields a quantity of the same family.** `160 kg / 2` is `Mass` (= 80 kg).

Rules 3 and 4 are the operations a lifter actually writes ("half my top set," "double the working weight"); rules 1 and 2 are what makes the universe closed under the operations rule 3 and 4 admit.

### 4.5.5 Rates and rate cancellation

A `<rate>` literal `NUM_LIT UNIT₁ "/" UNIT₂` has type `Rate(family(UNIT₁), family(UNIT₂))`. So `2.5 kg/week` is `Rate(Mass, Time)`; `0.25 kg/day` is also `Rate(Mass, Time)` and is treated as equal-typed to `2.5 kg/week` after canonicalization to `g/s`.

One arithmetic form involving rates is admitted:

- **Rate × Time → Quantity.** `2.5 kg/week * 4 week` is `Mass` (= 10 kg). This is the rule §4.4's `progress` and `goal lose / gain` semantics rely on to project a base value forward over a horizon. Mathematically it is dimensional cancellation; in the checker it is a single rule keyed on the operator `*` between an operand of type `Rate(D, Time)` and an operand of type `Quantity Time`. The symmetric form `Time × Rate` is also admitted.

No other rate arithmetic is allowed in P1 idiom — no rate + rate (it would require asserting same dimensional families on both sides; harmless to add later), no rate / rate (would produce a dimensionless `Number` and reopen the universe). This is the same minimalism the unit-arithmetic rules in §4.5.4 commit to.

### 4.5.6 Record kinds — name-equivalent, schema-fixed

The four record kinds — `Exercise`, `Meal`, `Workout`, `Routine` — are not anonymous structs. Each is identified by its kind keyword and has a schema fixed by §4.3:

| Kind | Schema (fields → types) | Introduced by | Identity |
|---|---|---|---|
| `Exercise` | `sets: Number, reps: Number, weight: Mass` | `<exercise-decl>` | label (`STRING_LIT`), scoped to enclosing workout/routine |
| `Meal` | one or more of `protein: Mass, carbs: Mass, fat: Mass` | `<meal-decl>` | label, scoped to athlete |
| `Workout` | list of `Exercise` or `<routine-call>` results, optional `progress` rate | `<workout-decl>` | label, scoped to athlete |
| `Routine` | parameter list `(IDENT: UNIT, ...)` + body of `Exercise`s | `<routine-decl>` | label, scoped to athlete |

Because the schema is part of the grammar, the type checker has no "missing field" or "extra field" case to report — those are syntax errors that never reach it. Its record-kind responsibilities reduce to: (a) every numeric/quantity slot has the right type after evaluating its expression (`weight: <quantity-typed expr>` etc.), and (b) every `<routine-call>` matches the declared routine's arity and per-parameter family (`UNIT` annotation in `<routine-params>` fixes the family of each argument's expected `Quantity` type).

This is the same "structured types pin their identity to a name and their shape to a declaration" approach Pascal and Ada take for records (Sebesta §6.7); FitLang earns simplicity here by *not letting users introduce new record kinds*, so the table above is exhaustive and the checker can hard-code one rule per kind.

### 4.5.7 `let` bindings — what is nameable as a value

`<let-decl>` introduces a new `IDENT` and binds it to the value of its RHS expression. The admissible RHS types are exactly the three categories the type checker tracks:

- `Number` — `let proteinPerKg = 1.8`
- `Quantity D` for any family D — `let workingWeight = 80 kg`, `let dailyKcal = 2800 kcal`
- `Rate(D₁, D₂)` — `let bulkRate = 0.25 kg/week`

Record kinds (`Exercise`, `Meal`, `Workout`, `Routine`) are **not** let-bindable. The domain already gives them names — exercises and meals and workouts and routines each get a `STRING_LIT` label at their declaration — so a `let bench = exercise "..." { ... }` would introduce a second name space competing with the label space of §4.6.1 for no payoff. Reusable parameterized exercise groups go through `routine` / `use`, not through `let`. This is Sebesta §5.8 *named constants* taken at its narrowest reasonable reading: name what is a value (`Number`, `Quantity`, `Rate` — things you do arithmetic on), do not name what is already a labelled declaration.

Each let-bound name's type is inferred from its RHS at binding time — there is no surface annotation. Inference is trivial because the three categories are syntactically distinguishable (`NUM_LIT` alone → `Number`; `NUM_LIT UNIT` → `Quantity (family UNIT)`; `NUM_LIT UNIT "/" UNIT` → `Rate(family, family)`; an `IDENT` reference takes its referent's type from the environment).

### 4.5.8 Type-checking algorithm (informative)

The checker is a single bottom-up pass over the AST keyed on node kind, producing for each `<expression>` node a type in `{ Number, Quantity D, Rate(D₁, D₂) }` and for each declaration node either a unit type "well-typed" or a typed diagnostic. Pseudocode for the expression core (the only nontrivial part — every other rule is a single shape check on a fixed schema):

```
check(NUM_LIT n)              = Number
check(QUANTITY n u)           = Quantity (family u)
check(RATE n u1 u2)           = Rate(family u1, family u2)
check(IDENT x)                = env(x)                      // single env, single pass
check(e1 "+" e2)
  | (Number, Number)          = Number
  | (Quantity D, Quantity D)  = Quantity D                  // same family, §4.5.4(1)
  | otherwise                 = error "cannot add T1 and T2" // rejects Rate+Rate, cross-family, mixed
check(e1 "-" e2)              = same as "+"
check(e1 "*" e2)
  | (Number, Quantity D)      = Quantity D                  // rule 4.5.4(3)
  | (Quantity D, Number)      = Quantity D
  | (Number, Number)          = Number
  | (Rate(D, Time), Quantity Time) = Quantity D             // rule 4.5.5
  | (Quantity Time, Rate(D, Time)) = Quantity D
  | otherwise                 = error
check(e1 "/" e2)              // only outside <rate> context
  | (Quantity D, Number)      = Quantity D                  // rule 4.5.4(4)
  | (Number, Number)          = Number
  | otherwise                 = error
```

The environment is the flat chain of `let`-bindings established by §4.6.3 — no scoping stack, no shadowing. E3 (D4) re-implements this checker via AI prompt and tests against ≥3 known-answer inputs.

## 4.6 Names, Binding, Scope, Lifetime **[P1]**

### 4.6.1 What is nameable

FitLang has **two disjoint name spaces**, separated at the lexical level so the parser never has to disambiguate:

| Name space | Lexical class | Introduced by | Used in |
|---|---|---|---|
| **Labels** | `STRING_LIT` | `workout "..." { ... }`, `exercise "..." { ... }`, `meal "..." { ... }`, `routine "..." (...) { ... }` | the `<name-list>` of `<schedule-stmt>`; the head of a `<routine-call>` (`use "..."`) |
| **Identifiers** | `IDENT` | `<let-decl>` (athlete scope); `<routine-params>` (routine scope, parameter names) | right-hand side of any `<expression>`; `<quantity>` slots inside the routine body (P2) |

Workouts, exercises, and meals are addressed by their human-readable **string labels** (`"Push Day"`, `"Bench Press"`); these are the names the lifter or dietitian writes in a notebook, and the surface syntax preserves that — there is no separate "give your workout an internal identifier" step. Only `let`-bound numeric or quantity values get an `IDENT`, and the closed-lexicon rule of §4.2.5 guarantees an `IDENT` can never collide with a reserved word, mode literal, or unit. Sebesta §5.2 lists separate name spaces (Common Lisp's function-vs-value distinction) as a deliberate technique to enlarge the usable vocabulary without ambiguity; FitLang takes the same option for the same reason.

The two name spaces never interact: a `<schedule-stmt>` resolves its `<name-list>` against the label space only, and an `<expression>` resolves an `IDENT` against the identifier space only. There is consequently no `f(x)`-vs-`f` ambiguity, no `(name)` hack to disambiguate one from the other.

### 4.6.2 Binding times

Every binding in a FitLang program is fixed by **static binding time** in the sense of Sebesta §5.4.1 — bindings are made at compile time and do not change at run time:

| Attribute | Bound at | Mechanism |
|---|---|---|
| Name → declaration site | Compile time (parse) | The single `<workout-decl>` / `<meal-decl>` / `<let-decl>` that introduces the name |
| Name → type | Compile time (type check, §4.5 in P2) | Inferred from the right-hand side; declared types are not part of the surface syntax |
| Name → value | Compile time (constant evaluation) for all `<let-decl>`s whose right-hand side is closed under §4.7's expressions; otherwise at the start of run time | Pure declarative evaluation — no input, no I/O, no side effects |
| Name → storage | N/A | There is no observable storage: the interpreter is free to inline every name (Sebesta §5.4.3.4 *static* category, broadened to all bindings) |

FitLang has no analogue of Sebesta §5.4.2's *dynamic* binding times — no late-bound types, no first-class environments, no `eval`. This is a deliberate restriction: the domain user writes a plan once and runs it once, and every error the interpreter can move to compile time is one fewer surprise during the lifter's training week.

### 4.6.3 Scope

Scope is **static (lexical) and block-structured**, demarcated by the `{ ... }` brackets of `<athlete-block>`, `<workout-decl>`, `<exercise-decl>`, and `<meal-decl>`. We follow the C/Java/Pascal lineage Sebesta surveys in §5.5.1 and reject dynamic scoping (§5.5.2 — Sebesta credits it to early Lisps and notes it makes static analysis effectively impossible) outright: the DSL has no first-class procedures and no use case that would justify the cost.

The scope nesting is shallow and fixed by the grammar:

```
athlete-block
├── (let-decls, target-decls, goal-rate-decls)   ← athlete scope
└── workout-decl
    ├── exercise-decl                             ← exercise scope
    └── progress-stmt
        meal-decl                                 ← meal scope
        (macro-decls)
```

Three concrete rules follow:

1. **`let`-bound `IDENT`s live at athlete scope and only at athlete scope.** The grammar admits `<let-decl>` only as a `<rule-decl>` inside `<rules-section>`, never inside a `<workout-decl>` or `<meal-decl>`. A let-binding is therefore visible in every `<expression>` that follows it within the same `<athlete-block>` and invisible outside that block (a moot point, since there is exactly one). This is Sebesta §5.7's *referencing environment* in its simplest possible form: a flat chain of one block.
2. **Labels live at athlete scope, partitioned by kind.** Within an `<athlete-block>` no two `<workout-decl>`s may share a `STRING_LIT`, no two `<meal-decl>`s may share one, and no two `<routine-decl>`s may share one — but a workout, a meal, and a routine *may* coexist with the same label (`workout "Recovery"`, `meal "Recovery"`, `routine "Recovery"`) because each is resolved against its own kind: `<schedule-stmt>` against workout/meal labels, `<routine-call>`'s `use "..."` against the routine label space. Exercise labels live at the *workout* scope (and at the *routine* scope inside `<routine-decl>`): `"Bench Press"` may appear in two different workouts as two distinct exercise declarations. Routine parameter names (`load`, `accessory_load`, ...) live at the *routine* scope and are visible only inside that routine's body. This matches Pascal's record-field rule (Sebesta §5.5.5) — field names are scoped to their record, not globally — and the standard procedure-parameter scoping Sebesta surveys in §5.5.
3. **Forward references are forbidden.** A `<let-decl>` may only reference `IDENT`s introduced by an earlier `<let-decl>`; a `<schedule-stmt>` may only reference labels introduced by earlier `<workout-decl>`s and `<meal-decl>`s. The grammar already places `<schedule-stmt>` last, so the schedule-side of the rule is automatic; the let-side is a static check. This rules out mutual recursion among `let` bindings, which is exactly what we want — there is no recursion in the domain, and the no-forward-reference rule lets the type checker run in a single linear pass over the AST.

**No shadowing.** Re-binding an `IDENT` already in scope is a static error. With immutable bindings and a single flat scope for `let`, shadowing buys nothing and only invites the "which `dailyKcal` did I mean?" class of bug Sebesta flags in §5.5.4. (This is stricter than C/Java, which permit inner scopes to shadow outer ones; we are at liberty to be stricter because FitLang's nesting is a fixed three levels deep.)

### 4.6.4 Lifetime

Because every binding is immutable (§4.6.6 below) and there are no first-class procedures, closures, or heap allocations, **lifetime equals scope**. Sebesta §5.4.3 distinguishes four storage-binding categories — static, stack-dynamic, explicit-heap-dynamic, implicit-heap-dynamic — and FitLang collapses to a single one: every binding behaves *as if* statically allocated for the duration of the program, because no observable behaviour can distinguish that strategy from any other when nothing is mutated and the program is one-shot. The interpreter is free to allocate let-values on the stack, intern them, or constant-fold them away; the language definition does not require the user to know which.

The payoffs are concrete and worth listing for §4.8's rationale:

- No dangling-reference class of bug (Sebesta §5.4.3.3, Pascal pointer hazards) — there are no references.
- No initialise-before-use rule to specify (§5.4.3.1) — the grammar forces declaration-before-use, and the value-binding is part of the declaration.
- No garbage collection story (§5.4.3.4) — no heap.

### 4.6.5 Referencing environment

At any program point *p*, the referencing environment is the set of names visible at *p* (Sebesta §5.7). For FitLang it is:

- **Inside `<athlete-params>`:** the closed lexicon of §4.2 only — no user bindings exist yet.
- **Inside `<rules-section>` at point *p*:** the closed lexicon plus all `IDENT`s introduced by `<let-decl>`s strictly preceding *p* in the same `<rules-section>`.
- **Inside `<schedule-stmt>`:** the label space — every `STRING_LIT` introduced as a `<workout-decl>` or `<meal-decl>` head in the enclosing `<athlete-block>`.
- **Inside a `<workout-decl>` or `<meal-decl>` body:** only the closed lexicon and the constants written at the call site. (The grammar admits no `IDENT` reference inside these blocks in P1 — `weight: 80 kg` takes a literal `<quantity>`, not an `<expression>`. This is a P1-scoping decision; §4.7 in P2 will broaden the relevant fields to `<expression>` and the referencing environment will then include the athlete-scope `let`s.)

The bodyweight value declared in `<athlete-params>` is *not* an ordinary `IDENT` and does not appear in the identifier name space — it is a fixed parameter referred to by the keyword phrase `of bodyweight` in `<target-decl>`. This is the ALGOL-style approach of giving privileged values a reserved-word handle rather than an ordinary identifier (Sebesta §5.2 *special words*), and it sidesteps the question of what to do if a user wrote `let bodyweight = 90 kg`.

### 4.6.6 Immutability and named constants

There is no assignment operator (§4.7's `=` in `<let-decl>` is binding, not assignment; `:` in record literals is field initialisation). Every name in FitLang is therefore a **named constant** in Sebesta's §5.8 sense — bound once, at the declaration, and referenced thereafter without ever being updated. This is the same single-assignment discipline Erlang enforces at the language level and Rust enforces by default (`let` without `mut`).

The choice is domain-driven: a fitness program is a *plan*, not a process. Mutating "Push Day" mid-program does not correspond to anything the lifter does in the gym — if the workout changes, that is a *different* workout, and the language should make the user say so by introducing a new declaration. The cost paid for this discipline is the inability to express iterative state ("week 1 weight, week 2 weight, ..."); §4.4's `progress` construct (P2 semantics) recovers that expressivity by computing successive weeks declaratively from a base value and a rate, rather than by mutation.

## 4.7 Expressions & Assignment **[P2]**

### 4.7.1 The expression layer in full

P1 admitted only `<expression> → <term> { ("+" | "-") <term> }` over `NUM_LIT`, `<quantity>`, and `IDENT`. P2 broadens the layer to the full arithmetic subset the type rules of §4.5 admit, restated as a precedence-stratified EBNF:

```ebnf
<expression>   → <add-expr>
<add-expr>     → <mul-expr> { ("+" | "-") <mul-expr> }
<mul-expr>     → <unary>    { ("*" | "/") <unary>    }
<unary>        → [ "-" ] <atom>
<atom>         → NUM_LIT [ UNIT [ "/" UNIT ] ]      // scalar, quantity, or rate
               | IDENT
               | "(" <expression> ")"
```

`<rate>` (introduced lexically in §4.3.2 for `<progress-stmt>` and `<goal-rate-decl>`) is admitted into the atom alternative here so that `2.5 kg/week` can appear in a `<let-decl>` RHS as well as in those statement positions. The parser distinguishes the rate atom (`NUM_LIT UNIT "/" UNIT`) from the binary-`/` form (`<unary> "/" <unary>` inside `<mul-expr>`) by looking one token past the `/`: if the next token is a `UNIT` and the preceding atom was a bare `NUM_LIT UNIT` pair, it is a rate atom; otherwise it is multiplicative division. This is the same one-token disambiguation Sebesta describes for the C `*` operator (multiplication vs dereference, §4.4) and is LL(1) under the look-ahead the recursive-descent parser already maintains.

### 4.7.2 Operator precedence and associativity

| Level (tight → loose) | Operators | Associativity | Sebesta analogue (§7.2) |
|---|---|---|---|
| 1 | unary `-` | right | C, Java, Pascal |
| 2 | `*`, `/` | left | C, Java |
| 3 | `+`, `-` (binary) | left | C, Java |
| 4 | `>`, `<`, `>=`, `<=`, `==`, `!=` | non-associative | Pascal (Sebesta §7.2 — chained comparisons disallowed) |

Relational operators appear only inside `<target-decl>` (`target <macro> <op> <quantity>` per §4.3.2), never inside `<expression>`; they are not part of the precedence-stratified grammar above. The "non-associative" call means `a < b < c` is not writable in FitLang because the grammar admits exactly one `<rel-op>` after `target <macro>` — and we keep it that way because the Python-style chained-comparison reading (`a < b ∧ b < c`) is the only sensible one and the lifter writes one relation per target line.

There are **no boolean operators** (`&&`, `||`, `!`) and consequently no short-circuit semantics question (Sebesta §7.7). The only conditional construct in the language, `<when-decl>`, dispatches on a single mode literal (§4.4.10) — there is no need for compound predicates. Adding boolean operators later would be a strictly additive change at precedence level 5 (below relational).

### 4.7.3 Operand evaluation order

Operand evaluation is **left-to-right** by convention — but the order is *unobservable*. By §4.6.6 immutability and §4.4.1 purity, two evaluation orders that respect the no-forward-reference rule produce identical values. The Sebesta §7.5 "functional side effects in operands" hazard (the classic `a + f(a)` problem in C) does not exist in FitLang because no expression form has side effects: `<let-decl>` is the only environment-extending construct and it is a statement, not an expression. We therefore do not specify a stricter operand order than left-to-right and the language definition does not constrain implementers who would prefer to constant-fold or reorder for performance.

### 4.7.4 Assignment statement — there isn't one

FitLang has **no assignment statement** in Sebesta §7.6's sense (an L-value target whose stored value is replaced). The two tokens that look like assignment in the surface syntax mean something else:

- `=` in `<let-decl>` (`let x = <expression>`) is the **binding** operator — it introduces a fresh name in the immediately enclosing environment and is illegal if the name is already bound (§4.6.3 no-shadowing). Semantically it is the `let x = e in ...` of ML, not the `x := e` of Pascal. There is no L-value here — `x` is a name, not a cell.
- `:` in record literals (`weight: 80 kg`, `protein: 40 g`) is the **field initialisation** operator — it pairs a fixed schema field name with its value at construction time. The schema is closed (§4.5.6) so this is not a generic key-value form; it is the same syntax Ada uses for aggregate component association (Sebesta §6.7) and Pascal for record-literal initialisation.

The choice to omit assignment entirely is downstream of §4.6.6 immutability (single-assignment discipline, in the Erlang / Rust-default sense). Assignment requires storage to mutate, storage requires the user to reason about *when* the cell holds *which* value, and that reasoning is exactly what Sebesta §7.6 spends three pages warning the reader about. FitLang sidesteps the problem by not paying for the expressivity in the first place — the iterative state a lifter cares about (week-over-week weight progression) is recovered declaratively by `progress` + `plan N week` (§4.4.6, §4.4.8), not by mutating a cell.

### 4.7.5 Expression type rules — summary

The type of every well-typed expression is determined by §4.5.4–§4.5.5. Relevant for §4.7's broader form:

- Unary `-` preserves the type of its operand (`Number → Number`, `Quantity D → Quantity D`, `Rate(D₁,D₂) → Rate(D₁,D₂)`).
- Parenthesised expressions take the type of their inner expression.
- `*` and `/` outside `<rate>` follow §4.5.4 rules 3–4 and §4.5.5; in particular, same-family `*` and `/` remain static errors, so the broadened expression layer of §4.7.1 does not enlarge the value set, only the surface forms that can produce it.

## 4.8 Design Rationale **[P2]**

This section restates the major design decisions as decisions — what was on the table, what was picked, and what was given up to pick it. Each paragraph names one trade-off Sebesta would recognise.

**Closed lexicon, no shadowing.** Every alphabetic lexeme is either a reserved word, mode literal, unit, or an `IDENT` disjoint from all three (§4.2.5, §4.6.3). We pay the rigidity Sebesta credits Pascal for (§3.1) — a user cannot use `weight` as a let-name — and we earn the property that the parser never has to disambiguate keyword from identifier and the type checker never has to resolve a shadowed binding. For a DSL whose users are not software engineers, the rigidity cost is invisible (no one wants to call their let-binding `weight`) and the predictability payoff is real (every error message can point at a single declaration site).

**Strong typing with dimensional families.** §4.5.2 commits to compile-time rejection of every cross-family operation and disallows same-family `*` and `/` outside the named `per UNIT of bodyweight` sugar. The cost is the ergonomic ceremony Sebesta credits Java for paying against C in §1.6 — `80 kg + 200` is rejected, the user writes `+ 200 g` (or `+ 0.2 kg`). The payoff is that the entire class of "I mixed kcal and kJ" or "I wrote a kg where I meant g" bugs is eliminated at parse-plus-check time, before the lifter sees a single nonsensical training week.

**Name equivalence on record kinds.** §4.5.3 picks name over structural equivalence. The cost is the loss of structural flexibility — a hypothetical future kind that happens to share `exercise`'s schema would not interoperate with it. The payoff is the one-sentence quiz defence ("same kind ⇒ same type") and a checker that needs no field-by-field comparison routine. Because the grammar pins each kind to one schema, name and structural agree on every legal program — the choice is *which rule to teach*, and name is the simpler teach.

**Single-assignment, no mutation.** §4.6.6 commits to immutability. The cost is the inability to express imperative iteration — `weight = weight + 2.5 kg` is not writable. The payoff is that lifetime = scope (§4.6.4), no dangling references, no initialise-before-use rule, no garbage collection story, and a single-pass type checker. The iterative content the domain needs (week-over-week weight) is recovered declaratively by `progress` + `plan N week`, which is honest to how a lifter actually writes a periodised block: a base, a rate, a horizon.

**Static (lexical) scoping with a flat athlete-scope env.** §4.6.3 takes Sebesta §5.5.1's C/Pascal/Java lineage and rejects dynamic scoping outright. The cost — none, given no first-class procedures — is genuinely zero in this language, and the payoff is that the referencing environment at any program point is computable from the AST alone. Routines (§4.4.7) are scoped at their declaration site, not at the call site, for the same reason: the lifter should never write a routine whose meaning depends on which workout it is called from.

**Multi-week schedule as a first-class horizon.** §4.4.8 makes `plan N week` the unit of output, not `plan week`. The cost is a slightly larger interpreter loop and longer D3 outputs. The payoff is that `progress` and `goal lose/gain` become load-bearing — the language earns its keywords. A snapshot-only DSL would carry those constructs as ornaments; a horizon-bearing DSL uses them to project a typed training block, which is the difference between a notebook syntax and a planning language.

**Static `when`.** §4.4.10 decides the `<when-decl>` branch at compile time because the dispatch value is the literal goal mode bound in `<athlete-params>`. The cost is the inability to predicate on runtime values — but FitLang has no runtime values to predicate on (no input). The payoff is zero dispatch cost and dead-branch elimination before the semantics rules run, which keeps the operational story uniform: every program is a straight-line evaluation.

**No user-defined types.** §4.5.1 freezes the type universe at three scalar-or-quantity categories plus four named record kinds. The cost is extensibility — a user cannot introduce a `circuit` kind without changing the grammar. The payoff is the single-pass linear-time checker (§4.5.2) and the closed equivalence rule (§4.5.3); both rely on the universe being finite and listable. Extensibility for a course DSL is not a real requirement, and the version that *does* want it can spend a paragraph adding `kind "circuit" { ... }` without disturbing anything else.

**Two name spaces (label vs identifier).** §4.6.1 separates `STRING_LIT` labels from `IDENT` let-names at the lexical level. The cost is a minor learning surface (a user has to know which kind of name goes where) — moot because the grammar slots each kind in different positions and the surface syntax makes the choice for them. The payoff is the Common Lisp §5.2 advantage Sebesta lists: an enlarged vocabulary with zero parser ambiguity, and the consequence that a `<schedule-stmt>` and an `<expression>` can never collide on the same name.

**Java tree-walking interpreter.** Implementation choice driven by Sebesta §1.7's pure-interpretation discussion — at least an order of magnitude slower than compiled code, irrelevant for the domain (a 78 kg lifter is not going to notice the difference between a 5 ms and a 50 ms plan calculation). The payoff is the simplest possible mapping from §4.4's operational rules to executable code, which is also what makes E3 (D4 — AI implements the type checker) a tractable experiment to evaluate.
